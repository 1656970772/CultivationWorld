# 三层关系底座重构规格

> 最后更新：2026-06-07  
> 状态：设计确认，待用户审阅  
> 适用范围：个体关系、群体评价、势力声望、关系事件写入、关系决策信号、现有 AI 决策系统接入  
> 关联：`docs/data-models/relationship.md`、`docs/worldbuilding/wiki/characters/relationship-types.md`、`docs/worldbuilding/relations.md`、`docs/worldbuilding/wiki/rules/faction-crisis-defection.md`、ADR-027、ADR-028、ADR-029、ADR-048、ADR-050

## 背景

当前项目已有世界级 `RelationshipSystem`，负责 NPC/NPC、人/妖、妖/妖的关系边；NPC 侧 `RelationshipGraph` 已作为兼容视图接入个人恩怨；关系边也已经能产出同门驰援、报恩、师徒互动等关系 Goal。

本轮讨论确认：后续关系系统不应成为新的行为裁决器，也不应绕开项目现有的 Reaction / Dynamic Goal / GOAP / Utility / Job-Toil 决策链。关系系统应作为“社会上下文层”，记录三层关系状态，并把这些状态转换为现有决策系统可消费的信号。

## 目标

1. 建立“个体-群体-势力”三层关系底座。
2. 三层都统一采用“核心属性 + 主观潜力预期 + 事件驱动 + 行为反馈信号”的结构。
3. 关系变化接入现有行为、事件、记忆、战斗、死亡、动态事件和世界规则，不另造一套孤立事件体系。
4. 关系系统输出 `RelationshipDecisionSignals`，作为现有决策系统的门槛、权重调制、目标偏好和上下文事实。
5. 明确关系系统只输出信号，不输出最终行为决策。

## 非目标

- 不新增独立 `Resolver` 或关系裁决器来决定 NPC 最终行为。
- 不为每个事件类型单独写决策器。
- 不一次性实现所有关系反馈行为。
- 不把势力外交矩阵强行并入个人关系图。
- 不让关系系统跳过生存判断、战力风险、资源限制、冷却、任务链和当前 AI 架构。
- 不在本轮重写现有 `RelationshipSystem` 的全部行为；后续实施需兼容现有关系边和存档。

## 总体原则

```text
关系系统 = 记录、更新、产出信号
决策系统 = 选择目标、选择行为、执行 Job/Toil
```

关系系统对决策系统应是“强影响，但不独裁”：

| 影响档位 | 中文名 | 含义 | 示例 |
|----------|--------|------|------|
| `soft_bias` | 轻影响 | 调高或调低候选行为权重 | 好感高使赠礼权重上升 |
| `strong_bias` | 强影响 | 显著改变优先级，但仍可被生存、战力、资源压过 | 救命恩显著提高救援权重 |
| `hard_gate` | 硬门槛 | 解锁或禁止候选行为 | 信任不足不生成分享秘密行为 |

默认取向：

| 行为类别 | 关系影响建议 |
|----------|--------------|
| 日常行为 | 10%~30% |
| 社交、门派、交易行为 | 30%~60% |
| 生死、背叛、血仇、救命恩 | 60%~90% |

生存本能、实力差距、资源约束和任务可执行性仍可压过关系信号。

## 三层账本

### 个体层：IndividualRelation

个体层表示“某个 NPC 如何看待另一个 NPC”。核心属性只设两个主轴：

| 英文 key | 中文名 | 含义 |
|----------|--------|------|
| `core.affinity` | 好感度 | 情感倾向，影响亲近、赠礼、帮腔、落井下石 |
| `core.trust` | 信任度 | 利益绑定和托付程度，影响守密、作证、共享资源 |

示例结构：

```js
IndividualRelation {
  fromNpcId,
  toNpcId,
  core: {
    affinity,
    trust
  },
  potential,
  marks,
  tags,
  lastUpdatedDay
}
```

设计解释：

- 好感 100、信任 0：愿意聊天、喝酒、送小礼，但不会借宗门秘宝。
- 信任 100、好感 50：平时态度冷淡，但被诬陷时会拿出关键证据。

### 群体层：GroupStanding

群体层表示“某个门派、家族、派系、任务小队如何评价这个 NPC”。

| 英文 key | 中文名 | 含义 |
|----------|--------|------|
| `core.identity` | 群体认同度 | 是否把此人当自己人 |
| `core.trust` | 群体信任度 | 是否认为此人可靠、守规矩、可托付 |

示例结构：

```js
GroupStanding {
  npcId,
  groupId,
  core: {
    identity,
    trust
  },
  potential,
  marks,
  tags,
  lastUpdatedDay
}
```

设计解释：

- 高认同、高信任：更容易被支援、提拔、保护。
- 高认同、低信任：被视为自己人，但重要任务或秘宝不轻易托付。
- 低认同、高潜力：可能被重点观察、拉拢，也可能被内部排挤。

### 势力层：FactionReputation

势力层表示“某个宗门、阵营、正邪道、妖族或商会体系如何公开评价这个 NPC”。

| 英文 key | 中文名 | 含义 |
|----------|--------|------|
| `core.reputation` | 势力声望 | 公开正面评价，影响邀请、赐封、任务开放 |
| `core.credit` | 势力信用 | 是否守约、可交易、可合作 |

示例结构：

```js
FactionReputation {
  npcId,
  factionIdOrCampId,
  core: {
    reputation,
    credit
  },
  potential,
  marks,
  tags,
  lastUpdatedDay
}
```

设计解释：

- 通缉、恶名、黑名单不放入核心属性，而作为事件标记表达。
- 正道、邪道、妖族、商会可以分别持有不同的公开评价。

## 主观潜力预期

三层都必须包含 `PotentialAssessment`。它记录的是“评价者如何判断此人的未来价值”，不是 NPC 的真实资质。

```js
PotentialAssessment {
  expectedPotential,
  confidence,
  strategicValue,
  threatValue,
  investmentIntent,
  suppressionIntent,
  lastEvaluatedDay
}
```

| 英文 key | 中文名 | 含义 |
|----------|--------|------|
| `expectedPotential` | 预期潜力 | 主观判断此人未来能走多远 |
| `confidence` | 判断置信度 | 亲眼见证、传闻、误判或偏见的可靠性 |
| `strategicValue` | 战略价值 | 是否值得拉拢、培养、保护或利用 |
| `threatValue` | 威胁价值 | 是否可能成为未来大敌 |
| `investmentIntent` | 投资意愿 | 是否倾向传功、赠宝、提拔、护道 |
| `suppressionIntent` | 打压意愿 | 是否倾向排挤、暗害、提前扼杀 |

潜力预期只调制候选行为，不凭空生成最终行为。高潜力可以提高招揽、培养、保护、打压权重，但仍需现有决策系统判断是否行动。

## 事件驱动结构

三层关系都采用“现有事件源 + 影响规则”的写入方式，不新增一套孤立核心事件列表。

```js
RelationImpactRule {
  sourceType,
  sourceId,
  visibility,
  layerImpacts: {
    individual,
    group,
    faction
  },
  marksAdded,
  tagsAdded,
  potentialSignals
}
```

| 字段 | 中文名 | 说明 |
|------|--------|------|
| `sourceType` | 来源类型 | `action`、`job`、`toil`、`memory`、`combat`、`death`、`dynamic_event`、`world_rule` |
| `sourceId` | 来源 ID | 现有行为、任务、记忆、事件或规则 ID |
| `visibility` | 可见范围 | `private`、`group`、`public` |
| `layerImpacts` | 分层影响 | 同一事件可同时影响个体、群体、势力 |
| `marksAdded` | 新增标记 | 救命恩、血仇、通缉令等 |
| `tagsAdded` | 新增标签 | 师徒、同门、培养对象等 |
| `potentialSignals` | 潜力信号 | 展现战力、越阶胜利、炼丹天赋等 |

可见范围决定写入层级：

- `private`：主要影响个体层。
- `group`：影响个体层和相关群体层。
- `public`：可影响个体层、群体层和势力层。

## 标记：RelationMark

标记表示由关键事件造成的长期状态。标记不等同于普通数值；某些标记可作为强门槛或强权重来源。

```js
RelationMark {
  type,
  weight,
  sourceType,
  sourceId,
  visibility,
  createdDay,
  decayRule,
  expiresDay,
  consumed
}
```

第一批标记字典：

## 行为反馈规则

行为反馈不由关系系统直接执行，而是转化为现有决策系统可用的候选行为门槛、权重和目标偏好。

```js
RelationFeedbackRule {
  layer,
  trigger,
  mode,
  actionId,
  priority,
  cooldownDays,
  consumesMarks
}
```

| 字段 | 中文名 | 说明 |
|------|--------|------|
| `layer` | 层级 | `individual`、`group`、`faction` |
| `trigger` | 触发条件 | core、potential、mark、tag、当前决策上下文 |
| `mode` | 反馈模式 | `active` 主动反馈、`passive` 被动反馈 |
| `actionId` | 行为 ID | 后续接现有 Action / Job / Toil |
| `priority` | 优先级 | 只作为信号，不直接抢占现有系统 |
| `cooldownDays` | 冷却天数 | 避免反馈行为频繁刷屏 |
| `consumesMarks` | 消耗标记 | 可消耗救命恩、人情债、誓约等 |

反馈类别：

| 模式 | 中文名 | 示例 |
|------|--------|------|
| `active` | 主动反馈 | 赠礼、传功、告密、邀请、招揽、托孤、暗害、雇佣杀手 |
| `passive` | 被动反馈 | 帮腔、作证、救援、压案、通风报信、落井下石、拒绝交易、执行追杀 |

## 决策信号

`RelationshipDecisionSignals` 是关系系统交给现有决策系统的唯一标准输出。它只表达信号，不表达最终决策。

```js
RelationshipDecisionSignals {
  gates,
  modifiers,
  targetPreferences,
  facts
}
```

### Gates：候选行为门槛

| 英文 key | 中文名 | 用途 |
|----------|--------|------|
| `gates.canShareSecret` | 可分享秘密 | 信任足够且无背叛标记时允许分享秘境或情报 |
| `gates.canRequestRareItem` | 可请求珍稀物 | 高信任或高人情债时允许借秘宝、要丹药 |
| `gates.canTriggerWantedHunt` | 可触发追杀 | 势力层存在通缉或通缉热度足够高 |
| `gates.canIntercede` | 可出面求情 | 个体高信任或救命恩，且群体内有支持度 |
| `gates.canInvest` | 可投资培养 | 潜力预期高，且不是强敌对状态 |

### Modifiers：权重调制

| 英文 key | 中文名 | 用途 |
|----------|--------|------|
| `modifiers.giftWeight` | 赠礼权重 | 提高或降低主动赠礼倾向 |
| `modifiers.rescueWeight` | 救援权重 | 提高或降低驰援、护短、救命倾向 |
| `modifiers.huntWeight` | 追杀权重 | 提高或降低追杀、复仇、执行通缉倾向 |
| `modifiers.slanderWeight` | 中伤权重 | 提高或降低背后中伤、添油加醋倾向 |
| `modifiers.investWeight` | 投资权重 | 提高或降低培养、提拔、赠宝倾向 |
| `modifiers.intercedeWeight` | 求情权重 | 提高或降低压案、保人、通风报信倾向 |
| `modifiers.tradeAccessWeight` | 交易准入权重 | 提高或降低开放交易、封锁交易倾向 |

### Target Preferences：目标偏好

| 英文 key | 中文名 | 用途 |
|----------|--------|------|
| `targetPreferences.rescueTargetBonus` | 救援目标加成 | 让救援行为优先选择深关系对象 |
| `targetPreferences.revengeTargetBonus` | 复仇目标加成 | 让复仇行为优先选择血仇对象 |
| `targetPreferences.giftTargetBonus` | 赠礼目标加成 | 让赠礼行为优先选择高好感或高投资价值对象 |
| `targetPreferences.recruitTargetBonus` | 招揽目标加成 | 让招揽行为优先选择高战略价值对象 |
| `targetPreferences.suppressTargetBonus` | 打压目标加成 | 让打压行为优先选择高威胁价值对象 |

### Facts：上下文事实

| 英文 key | 中文名 | 用途 |
|----------|--------|------|
| `facts.hasLifeDebt` | 有救命恩 | 决策系统可用于救援、压案、求情判断 |
| `facts.hasBloodFeud` | 有血仇 | 决策系统可用于复仇、追杀、投敌判断 |
| `facts.isWantedByFaction` | 被势力通缉 | 决策系统可用于追杀、交易封锁、审查 |
| `facts.groupTrustLow` | 群体信任低 | 决策系统可用于审查、排挤、降权 |
| `facts.targetPotentialHigh` | 目标潜力高 | 决策系统可用于培养、招揽、打压 |
| `facts.relationExposure` | 关系公开程度 | `private`、`group`、`public`，影响公开或私下行动 |

示例输出：

```js
{
  gates: {
    canShareSecret: true,
    canTriggerWantedHunt: true,
    canIntercede: true
  },
  modifiers: {
    rescueWeight: 2.5,
    huntWeight: 0.6,
    investWeight: 1.8,
    intercedeWeight: 2.0
  },
  targetPreferences: {
    rescueTargetBonus: 80,
    revengeTargetBonus: 0,
    recruitTargetBonus: 70
  },
  facts: {
    hasLifeDebt: true,
    hasBloodFeud: false,
    isWantedByFaction: true,
    targetPotentialHigh: true,
    relationExposure: "group"
  }
}
```

## 与现有 AI 的接入

关系系统只提供以下四类输入：

1. `gates`：决定某些候选行为能否进入现有行为池。
2. `modifiers`：调制 Utility / GOAP 候选行为权重。
3. `targetPreferences`：影响同一行为的目标选择。
4. `facts`：作为 Reaction、Dynamic Goal、GoalProvider、Job/Toil 的上下文事实。

接入流程：

```text
现有事件/行为/记忆
  -> RelationImpactRule 更新三层账本
  -> RelationshipSignalProvider 提取信号
  -> 现有 AI 决策链消费：
       Reaction：紧急刺激
       Dynamic Goal / GoalProvider：生成候选目标
       GOAP：行为门槛和目标状态
       Utility：权重调制
       Job-Toil：执行具体行为
```

当前已有关系 Goal 的路径后续可保留，但语义应调整为“关系系统产出候选目标和信号，最终是否执行仍由 Planner / Utility / Job-Toil 决定”。

## 第一阶段范围

第一阶段只定义底座并为后续实现设边界：

1. 三层账本结构：`IndividualRelation`、`GroupStanding`、`FactionReputation`。
2. 三层通用结构：`PotentialAssessment`、`RelationMark`、`RelationTag`。
3. 事件写入规格：`RelationImpactRule`，来源接现有行为和事件。
4. 决策信号规格：`RelationshipDecisionSignals`。
5. AI 接入边界：关系系统只输出 signal / gate / modifier / target preference / fact。

第一阶段不要求立即实现所有反馈行为，但需要确保规格能解释“追杀令链”：

```text
势力层：wantedOrder / wantedHeat 提高追杀相关行为权重。
群体层：group.identity / group.trust 调低或调高组织执行意愿。
个体层：lifeDebt / trust 调高通风报信、压案、放水相关行为权重。
个体层：bloodFeud 调高追杀、暗害、投敌复仇权重。
现有决策系统：根据这些 signal 选择具体 Action / Job / Toil。
```

## 后续阶段建议

| 阶段 | 目标 |
|------|------|
| 阶段 2 | 兼容或迁移现有 `RelationshipSystem` 边到 `IndividualRelation` 语义 |
| 阶段 3 | `GroupStanding` 接宗门贡献、同门支援、危亡抉择 |
| 阶段 4 | `FactionReputation` 接通缉、正邪声望、交易封锁 |
| 阶段 5 | 更多主动/被动反馈接入 Job/Toil |

## 验证要求

后续实施时，验证不能依赖指纹或哈希。应通过真实模拟和直接行为统计观察：

1. 三层账本能被现有事件写入，且不破坏现有关系边读取。
2. 关系信号只影响候选行为、权重、目标和上下文，不直接执行行为。
3. 关闭新开关后，现有关系系统和 AI 主路径保持可回退。
4. 追杀令链能在真实模拟中表现出不同倾向：正常执行、放水、通风报信、加码追杀。
5. 高潜力目标在群体和势力层能产生培养、招揽或打压倾向，但仍受资源、战力、冷却和现有 AI 约束。

## 设计结论

三层关系底座的核心结论是：

```text
关系系统是决策系统的社会上下文层，不是新的 AI 大脑。
```

后续所有实现都应围绕这一边界展开：关系记录世界中的人情、群体评价和公共声望；现有 AI 决策系统根据这些社会信号决定 NPC 最终做什么。
