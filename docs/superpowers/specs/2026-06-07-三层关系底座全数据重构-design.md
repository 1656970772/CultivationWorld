# 三层关系底座全数据重构规格

> 最后更新：2026-06-07  
> 状态：最新设计已确认，进入分支实施  
> 前置讨论版本：`docs/superpowers/specs/2026-06-07-三层关系底座重构-design.md`  
> 适用范围：新三层关系底座、全数据关系规则、关系事件 Hook、关系影响规则、关系决策信号、旧关系系统删除与空壳兼容  
> 首个验收闭环：追杀令链
> 实施计划：`docs/superpowers/plans/2026-06-07-三层关系底座全数据重构实施计划.md`

## 背景

现有关系系统已经实现世界级 `RelationshipSystem`、NPC 侧兼容 `RelationshipGraph`、关系 Goal、师徒互动、同门驰援、报恩、妖群与领地关系等功能。但这些实现仍以旧边模型为核心：`RelationEdge { fromId, toId, type, affinity, strength }`，难以自然承载三层关系中的群体评价、势力声望、主观潜力、长期标记、身份标签与场景化决策信号。

本轮讨论确认：这次重构不再以兼容旧关系行为为第一优先级，而是先建立干净、可扩展、低耦合的新底座。旧关系行为可以删除或暂时失效；旧接口仅保留空壳，避免框架搭建阶段因调用点未清理而崩溃。

## 目标

1. 建立新的“个体-群体-势力”三层关系底座，作为未来唯一关系真相源。
2. 关系影响采用全数据模式：代码只提供通用解释器，业务关系逻辑全部写入配置。
3. 行为和事件只声明 `relationEventHooks`，不直接写三层账本。
4. 通过 `RelationImpactEngine` 读取配置，将标准 `RelationEvent` 转换为三层账本变化。
5. 通过 `RelationshipSignalProvider` 输出 `facts / gates / modifiers / targetPreferences`，由现有 AI 决策链消费。
6. 第一版以“追杀令链”验证三层底座能影响 NPC 决策、群体行为和势力需求。

## 非目标

- 不保留旧 `RelationshipSystem` 的边驱动行为作为新底座的一部分。
- 不在战斗、死亡、势力、任务、社交等领域代码里写死具体关系业务逻辑。
- 不在第一版复刻旧师徒互动、同门驰援、报恩、妖群护群、领地防御等旧行为。
- 不在第一版实现 trade、master-disciple、monster/beast、temporary-group 等全部规则域。
- 不让关系系统绕过现有 Reaction / Dynamic Goal / Utility / GOAP / Job-Toil / Faction Need 决策链。

## 总体架构

整体链路：

```text
行为 / 事件
  -> relationEventHooks
  -> RelationEventEmitter 生成标准 RelationEvent
  -> RelationImpactEngine 匹配 impact-rules
  -> SelectorResolver 解析 subject/object/group/faction 集合
  -> ExpressionEvaluator 计算数值
  -> LedgerRepository 写三层账本
  -> SignalProvider 按 signal-rules 输出 facts/gates/modifiers/targetPreferences
  -> NPC Goal/Action、TargetPreference、Faction Need 消费信号
```

核心原则：

```text
代码只提供通用解释器和账本读写原语。
业务关系逻辑全部由 apps/game/data/relationships/ 配置表达。
任何行为、战斗、死亡、势力系统不得直接写三层账本。
```

设计模式映射：

| 模式 | 用途 |
|------|------|
| 事件总线 / 发布订阅 | 行为、事件与关系系统解耦，只发布标准 `RelationEvent` |
| 解释器模式 | 解释 JSON 条件、选择器、表达式和效果 |
| 策略模式 + 注册表 | 扩展 selector operator、condition operator、effect operator |
| Pipeline / 责任链 | 事件标准化、规则匹配、对象解析、表达式计算、账本写入、trace 记录 |
| 仓储模式 | `LedgerRepository` 统一读写三层账本、快照和恢复 |
| 依赖倒置 | NPC、势力、战斗等领域只依赖 `RelationEventPort` / `SignalProvider` 接口 |

### 最新实施架构补充

本次实施按“端口 + 仓储 + 解释器 + 信号提供者 + 兼容门面”拆分，不再把旧边模型继续扩写成万能类。

| 模块 | 位置 | 职责 | 设计模式 |
|------|------|------|----------|
| `RelationshipSystem` 门面 | `apps/game/js/engine/world/relationship-system.js` | 保留旧调用点所需 API，内部委托新三层平台；旧 `addEdge/applyEvent/edgesOfType` 只作为兼容投影 | Facade / Adapter |
| `LedgerRepository` | `apps/game/js/engine/relationship/ledger-repository.js` | 三层账本唯一读写入口，负责 key、默认值、mark/tag 叠加、快照、衰减、清理 | Repository |
| `RelationEventEmitter` | `apps/game/js/engine/relationship/relation-event-emitter.js` | 把旧事件名、行为 hook 或显式事件转换为标准 `RelationEvent` | Adapter |
| `RelationImpactEngine` | `apps/game/js/engine/relationship/impact-engine.js` | 匹配数据规则并按 pipeline 写账本，不持有业务特例 | Interpreter / Pipeline |
| `SelectorResolver` | `apps/game/js/engine/relationship/selector-resolver.js` | 从事件、实体、账本上下文解析单体或集合 selector | Strategy + Registry |
| `ExpressionEvaluator` | `apps/game/js/engine/relationship/expression-evaluator.js` | 只允许白名单函数和无副作用表达式 | Interpreter |
| `EffectOperatorRegistry` | `apps/game/js/engine/relationship/effect-operators.js` | 注册 core、potential、mark、tag、consume 等 effect operator | Strategy + Registry |
| `RelationshipSignalProvider` | `apps/game/js/engine/relationship/signal-provider.js` | 按数据规则输出 facts/gates/modifiers/targetPreferences/traces | Interpreter |

依赖方向固定为：

```text
NPC / Faction / Combat / Tick
  -> RelationEventPort / SignalProvider
  -> RelationshipSystem facade
  -> ImpactEngine / LedgerRepository / SignalProvider
```

上层系统不得直接导入 `LedgerRepository` 或修改账本内部结构。新增关系业务优先新增 `apps/game/data/relationships/` 配置；只有当 selector、condition 或 effect 语法本身缺能力时，才新增 operator。

### 第一版实现边界

第一版实现“追杀令链”所需的最小可运行平台：

1. 三层账本、mark/tag 字典、事件类型字典、signal key 字典和规则加载校验。
2. `combat.kill`、`combat.damage`、`social.life_debt`、`social.blood_feud`、`faction.wanted`、`faction.wanted_revoked` 的标准事件和影响规则。
3. `wantedOrder`、`bloodFeud`、`lifeDebt`、`sameSect`、`masterDisciple` 的兼容投影，保证旧调用点不崩。
4. `RelationshipSignalProvider.getFacts/getSignals` 输出通缉、血仇、救命恩、追杀权重、放水/求情倾向。
5. NPC 侧只消费信号：根据势力通缉和个体血仇更新 `hasRevengeTarget`/目标偏好；根据救命恩或高信任降低追杀权重或产生通风报信/求情候选信号。最终行为仍由 GOAP / Utility / Job-Toil 裁决。

旧师徒互动、妖群、领地威胁等旧边行为第一版只保留兼容读写，不再作为新底座的设计证明。后续要恢复这些行为时，必须迁成 tag/mark/signal rule，而不是恢复旧边模型。

## 三层账本

三层账本的语义：

```text
IndividualRelation：subject 如何看 object
GroupStanding：group 如何评价 subject
FactionReputation：faction 如何公开评价 subject
```

底层 ID 泛化，账本保存 `subjectType/objectType`，未来可接 NPC、妖兽、势力实体、组织实体。第一版只激活 NPC 社会关系。

### IndividualRelation

```js
IndividualRelation {
  layer: "individual",
  subjectId,
  subjectType,
  objectId,
  objectType,
  core: {
    affinity,
    trust
  },
  potential,
  marks,
  tags,
  lastUpdatedDay
}
```

### GroupStanding

```js
GroupStanding {
  layer: "group",
  groupId,
  subjectId,
  subjectType,
  core: {
    identity,
    trust
  },
  potential,
  marks,
  tags,
  lastUpdatedDay
}
```

Group 第一版采用中定义：稳定组织和稳定子群体才有持久账本。包括宗门、家族、商会、杀手组织、长老会、派系、长期任务队。秘境临时队、围观群众、临时同盟只作为事件上下文，宽定义 group 标记为后续待做。

### FactionReputation

```js
FactionReputation {
  layer: "faction",
  factionId,
  subjectId,
  subjectType,
  core: {
    reputation,
    credit
  },
  potential,
  marks,
  tags,
  lastUpdatedDay
}
```

`FactionReputation` 是“某势力/阵营如何公开评价某个主体”，不等同于现有势力间 `faction.state.relations` 外交矩阵。

## PotentialAssessment

三层账本共用主观潜力预期：

```js
PotentialAssessment {
  expectedPotential,
  confidence,
  strategicValue,
  threatValue,
  investmentIntent,
  suppressionIntent,
  lastEvaluatedDay
}
```

潜力预期是评价者的主观判断，不是 NPC 真实资质。它只输出培养、招揽、保护、打压等决策信号，不直接决定最终行为。

## Mark 与 Tag

`mark` 和 `tag` 必须严格区分：

```text
mark：事件造成的长期状态，有 weight、来源、衰减、消耗、可见性，可多实例。
tag：身份或结构性关系解释，有来源、有效期和 active 状态，通常不消耗。
```

### RelationMark

```js
RelationMark {
  id,
  type,
  weight,
  source: {
    eventId,
    ruleId,
    sourceType,
    sourceId
  },
  visibility,
  createdDay,
  expiresDay,
  decayRule,
  consumable,
  consumed
}
```

Mark 字典示例：

```json
{
  "id": "wantedOrder",
  "name": "通缉令",
  "layers": ["faction"],
  "stacking": "max",
  "defaultWeight": 50,
  "maxWeight": 100,
  "decay": { "type": "linear", "value": 0.1 },
  "consumable": false,
  "signalHints": ["huntWeight", "tradeAccessWeight"]
}
```

支持的 `stacking`：

| stacking | 说明 |
|----------|------|
| `unique` | 只保留一个 |
| `max` | 保留最高 `weight` |
| `add` | 累加到 `maxWeight` |
| `refresh` | 刷新有效期/来源 |
| `instance` | 保留多条实例，如多笔人情债 |

### RelationTag

```js
RelationTag {
  id,
  type,
  source: {
    eventId,
    ruleId,
    sourceType,
    sourceId
  },
  visibility,
  createdDay,
  expiresDay,
  active,
  modifiers
}
```

Tag 字典示例：

```json
{
  "id": "sameSect",
  "name": "同门",
  "layers": ["individual", "group"],
  "stacking": "unique",
  "defaultModifiers": {
    "affinityFloor": 10,
    "trustDecayMult": 0.8
  }
}
```

## 配置目录

新配置统一放在：

```text
apps/game/data/relationships/
```

建议目录：

```text
apps/game/data/relationships/
  schemas/
    ledgers.json
    event-schema.json
    expression-context.json

  dictionaries/
    marks.json
    tags.json
    signal-keys.json
    relation-event-types.json
    group-types.json

  event-hooks/
    npc-actions.json
    faction-actions.json
    combat-events.json
    death-events.json
    dynamic-events.json

  impact-rules/
    combat.json
    death.json
    faction.json
    group.json
    social.json

  signal-rules/
    npc-goals.json
    npc-actions.json
    target-preferences.json
    faction-needs.json

  groups/
    groups.json
```

配置职责：

| 目录 | 职责 |
|------|------|
| `schemas/` | 定义三层账本字段、数值范围、事件结构、表达式上下文白名单 |
| `dictionaries/` | 定义 mark/tag/signal/event/group 类型，不允许裸字符串散落 |
| `event-hooks/` | 把现有 action/event 映射成标准 `RelationEvent` |
| `impact-rules/` | `RelationEvent` 如何写入三层账本 |
| `signal-rules/` | 账本如何输出 AI 可消费信号 |
| `groups/` | 稳定组织/子群体定义和成员来源规则 |

旧的 `apps/game/data/balance/relationship.json` 后续废弃或仅保留迁移说明，不再作为新三层底座主配置。

## RelationEvent Hook

行为/事件配置只声明 hook：

```json
{
  "id": "act_npc_kill_enemy",
  "relationEventHooks": ["combat.kill"]
}
```

关系配置中定义 hook 模板：

```json
{
  "hook": "combat.kill",
  "event": {
    "type": "combat.kill",
    "actor": "$entity.id",
    "target": "$result.targetId",
    "source": {
      "type": "action",
      "id": "$action.id"
    },
    "visibility": "$result.visibility",
    "witnesses": "$result.witnesses",
    "location": "$entity.position"
  }
}
```

这让行为文件保持简洁，同时仍明确声明自身会触发哪些关系事件。

## Impact Rule

标准结构：

```json
{
  "id": "combat_kill_public_wanted",
  "priority": 50,
  "exclusiveGroup": "kill_severity",
  "match": {
    "eventType": "combat.kill",
    "visibility": ["group", "public"],
    "conditions": [
      { "expr": "target.factionId != null" },
      { "expr": "actor.id != target.id" }
    ]
  },
  "effects": [
    {
      "layer": "faction",
      "faction": "$target.factionId",
      "subject": "$actor.id",
      "changes": {
        "core.reputation": {
          "delta": {
            "expr": "-clamp(target.roleRank * 8 + witness.count * 2, 20, 80)"
          }
        },
        "marks": [
          {
            "type": "wantedOrder",
            "weight": {
              "expr": "clamp(target.roleRank * 12 + witness.count * 2, 20, 100)"
            }
          }
        ]
      }
    }
  ]
}
```

规则匹配机制：

```text
默认所有命中规则叠加。
同一 exclusiveGroup 内只取 priority 最高者。
不同 exclusiveGroup 互不影响。
```

Effect 对象字段使用 `subject/object`：

```text
individual：subject 如何看 object
group：group 如何评价 subject
faction：faction 如何公开评价 subject
```

## Selector

Selector 从 RelationEvent 上下文里解析影响对象，可返回单体或集合。

常见 selector：

```json
"$actor.id"
"$target.id"
"$target.factionId"
"$target.faction.members"
"$event.witnesses"
"$group.members"
"$ledger.marks[type=bloodFeud]"
```

第一版完整支持集合 selector，不提前限制表达力。性能优化后置，但每条规则必须记录：

```text
selector 命中数量
实际影响数量
执行耗时
是否被后续优化策略截断
```

## Expression DSL

表达式只做数值或布尔计算，不允许副作用。

标准上下文命名：

| 名称 | 说明 |
|------|------|
| `actor` | 行为发起者 |
| `target` | 行为直接目标 |
| `subject` | 当前 effect 解析出的被评价对象 |
| `object` | 当前 effect 解析出的关系对象 |
| `source` | 行为/事件来源 |
| `witness` | 见证信息 |
| `group` | 当前群体上下文 |
| `faction` | 当前势力上下文 |
| `ledger` | 当前命中的账本 |
| `world` | 世界公共信息 |

允许事件别名，如 `killer/victim`，但底层统一映射到 `actor/target`。

表达式示例：

```json
{
  "expr": "clamp(target.roleRank * 12 + witness.count * 2, 20, 100)"
}
```

第一版允许函数：

```text
clamp(value, min, max)
min(a, b)
max(a, b)
abs(value)
round(value)
floor(value)
ceil(value)
scale(value, fromMin, fromMax, toMin, toMax)
hasTag(entityOrLedger, tag)
hasMark(ledger, markType)
markWeight(ledger, markType)
markCount(ledger, markType)
```

表达式不能直接查询世界集合。集合查询只走 selector，避免表达式变成无限制脚本。

## 规则执行 Pipeline

```text
1. RelationEventEmitter 从 hook 生成标准事件
2. ImpactRuleMatcher 找到 eventType/source/visibility/conditions 命中的规则
3. ExclusiveGroupResolver 对互斥组取最高 priority，其余规则叠加
4. SelectorResolver 展开 subject/object/group/faction
5. LedgerRepository 找到或创建对应账本
6. ExpressionEvaluator 在当前上下文计算 delta/weight/condition
7. EffectOperator 执行 core/potential/mark/tag/consume/decay 等变化
8. TraceRecorder 记录解释链
```

错误处理：

```text
配置加载期校验：未知 mark/tag/signal/eventType、未知字段、表达式语法错误直接报错。
运行期规则错误：当前规则跳过并记录 trace，不中断整个 tick。
数值写入统一 clamp 到 schema 范围。
集合 selector 为空不是错误，只记录 no_targets。
```

## SignalProvider

账本不直接决定行为，只输出 `RelationshipDecisionSignals`。

```js
RelationshipDecisionSignals {
  facts,
  gates,
  modifiers,
  targetPreferences,
  traces
}
```

信号分两类：

```text
facts：全局稳定事实，任何系统可查。
gates / modifiers / targetPreferences：必须带 decisionContext 查询。
```

调用示例：

```js
signalProvider.getFacts({
  actorId: "npc_a",
  targetId: "npc_b"
})
```

```js
signalProvider.getSignals({
  actorId: "npc_a",
  targetId: "npc_b",
  contextType: "action",
  actionId: "act_npc_hunt_wanted"
})
```

Signal rule 示例：

```json
{
  "id": "wanted_hunt_signal",
  "scope": "contextual",
  "appliesTo": {
    "contextType": "action",
    "actionId": "act_npc_hunt_wanted"
  },
  "match": {
    "ledgers": [
      {
        "layer": "faction",
        "faction": "$actor.factionId",
        "subject": "$target.id"
      }
    ],
    "conditions": [
      { "expr": "hasMark(ledger, 'wantedOrder')" }
    ]
  },
  "outputs": {
    "facts": {
      "isWantedByFaction": true
    },
    "gates": {
      "canTriggerWantedHunt": true
    },
    "modifiers": {
      "huntWeight": {
        "expr": "1 + markWeight(ledger, 'wantedOrder') / 40"
      }
    },
    "targetPreferences": {
      "revengeTargetBonus": {
        "expr": "markWeight(ledger, 'wantedOrder')"
      }
    }
  }
}
```

## AI 接入

第一版固定接入点：

| 接入点 | 关系信号用途 |
|--------|--------------|
| NPC `collectExtraGoals` | 生成追杀、通风报信、求情、放水等候选 Goal |
| NPC `decorateGoalConsiderations` | 用 modifiers 调制 Goal score |
| GOAP action preconditions | 用 gates 控制行为能否进入计划 |
| target resolver / selector | 用 targetPreferences 影响同类行为优先选谁 |
| Faction need evaluation | 让 wantedHeat、声望、信用等影响通缉、封锁、庇护、追杀事务优先级 |

AI 仍负责最终选择。关系系统只提供社会上下文信号，不执行行为、不抢占生存判断、不绕开战力风险、资源限制、冷却和 Job/Toil 执行链。

## 第一版规则域

必须实现：

1. `combat/death`：击杀、伤害、救援、被攻击、死亡。
2. `faction`：通缉、撤销通缉、交易封锁、庇护名单。
3. `group`：贡献、违纪、同门评价、危亡抉择信号。
4. `social`：救命恩、人情债、背叛、血仇、求情、通风报信。

延后：

1. `trade`：信用、违约、商会 VIP、黑名单。
2. `master-disciple`：师徒传承、遗志、夺舍。
3. `monster/beast`：妖兽、人妖、族群。
4. `temporary-group`：临时小队、围观群众持久账本。

## 追杀令链验收

第一版用追杀令链验证三层底座：

```text
公开击杀或严重冒犯
  -> impact-rules 写 faction wantedOrder / wantedHeat
  -> signal-rules 让势力产生追杀/封锁倾向
  -> 群体层根据 identity/trust 决定成员执行强度或放水倾向
  -> 个体层根据 bloodFeud/lifeDebt/trust 产生追杀、通风报信、求情、放水候选
  -> AI 最终仍由现有 Utility / GOAP / Job-Toil / Faction Need 决定执行
```

验收现象：

| 现象 | 说明 |
|------|------|
| 正常执行 | 与目标无私交、群体信任高、通缉权重大，成员积极追杀 |
| 放水 | 群体执行但个体对目标有高信任或救命恩，执行权重降低 |
| 通风报信 | 个体层有 `lifeDebt` / 高 `trust`，生成告警或通风报信候选 |
| 加码追杀 | 个体层有 `bloodFeud`，在势力通缉之外提高追杀目标偏好 |
| 求情/压案 | 高信任或救命恩叠加群体身份，生成求情或降低处罚信号 |

## 旧关系系统处理

重构策略：

```text
删除旧关系行为，保留旧接口空壳。
```

处理原则：

1. 旧 `RelationshipSystem` 的边驱动 NPC Goal、师徒互动、妖群/领地关系逻辑不再作为新底座的一部分。
2. 旧 `RelationshipGraph` / `RelationshipSystem` API 可保留最小空壳，返回空集合或空值，避免未清理调用点崩溃。
3. 新系统稳定后，逐步删除旧 API 调用点。
4. 旧 `same_sect/master/grudge/gratitude/enemy` 语义未来分别迁入 tag、mark、core 或 signal rules，不再保留旧边模型。

## 验证要求

验证不能依赖指纹或哈希。应通过真实模拟和直接行为统计观察。

测试分层：

1. 配置加载校验：
   - mark/tag/signal/eventType 是否存在。
   - schema 字段范围是否合法。
   - 表达式语法是否合法。
   - selector 路径是否可解析或可延迟解析。
2. 规则引擎单测：
   - match / exclusiveGroup / priority。
   - selector 单体与集合。
   - expr 计算与白名单函数。
   - effect 写 core / potential / marks / tags。
   - mark stacking: `unique/max/add/refresh/instance`。
3. 账本仓储测试：
   - individual/group/faction key 唯一性。
   - snapshot/load。
   - decay/expire/consume。
   - trace 记录。
4. SignalProvider 测试：
   - facts 全局输出。
   - contextual gates/modifiers/targetPreferences。
   - 多条规则合并。
   - 无账本/无目标时安全返回。
5. AI 接入测试：
   - NPC Goal 生成追杀、通风报信、求情候选。
   - target preference 改变候选目标排序。
   - faction need 因 wantedHeat 提升通缉事务优先级。
   - gates 能阻止信任不足、无通缉的行为。
6. 真实模拟验证：
   - 多种子长程模拟。
   - 观察追杀令链出现：执行、放水、通风报信、加码追杀。
   - 输出行为统计和事件样本。

## 设计结论

本次重构的核心结论是：

```text
新三层关系系统是一个全数据社会关系规则平台。
代码提供解释器；配置定义关系业务；AI 消费信号而不被关系系统替代。
```

这能让“NPC 通过各种关系影响决策、影响群体行为、影响势力”成为可扩展底座，而不是一组孤立特例。
