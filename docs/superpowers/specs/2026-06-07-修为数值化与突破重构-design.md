# 修为数值化与突破重构规格

> 最后更新：2026-06-07  
> 状态：设计确认，待实施计划  
> 适用范围：NPC 修为字段、闭关/历练修为获取、境界小层推导、突破成功/失败结算、AI 行为收益评估  
> 关联：ADR-011、ADR-017、ADR-040、ADR-047、ADR-051、ADR-053

## 背景

当前修炼系统已经引入 `cultivation`、`experienceCultivation`、`totalCultivation` 等数值字段，但运行时仍保留 `cultivationProgress`、`insight`、`totalProgress`、`cultivationProgressRatio` 等 0..1 比例字段作为兼容或派生语义。这样会让修为、真气、历练收益、GOAP 目标和 UI 文案同时存在两套表达，后续接入功法感悟、小层、瓶颈和 12 境界时容易继续分叉。

本规格目标是把“修炼进度”彻底改为和真气一样的明确数值资源。本轮不扩展运行时境界链到 12 个大境界；战斗属性表中的 12 境界仍作为战斗参考，高阶运行时境界另行设计。

## 目标

1. 彻底移除旧比例字段，不再让业务逻辑、快照、UI、报告或测试依赖 `cultivationProgress`、`insight`、`totalProgress`、`cultivationProgressRatio`。
2. 使用数值修为作为唯一修炼进度口径：`cultivation`、`experienceCultivation`、`totalCultivation`。
3. `cultivationRequired` 暂时与目标境界的 `qiRequired` 相同，使修为门槛和真气门槛同量级。
4. 小层由 `totalCultivation / cultivationRequired` 推导，而不是独立进度条。
5. 闭关不再有硬上限，但闭关修为收益会随 `cultivation / cultivationRequired` 递减。
6. 历练、任务、斩妖、动态事件等外出行为继续提供 `experienceCultivation`，收益快于纯闭关但伴随风险。
7. 突破失败按明确数值惩罚结算：真气掉 50%，修为掉 80%，受重伤。

## 非目标

- 不在本轮扩展 `ranks.json` 到 12 个大境界。
- 不实现功法感悟系统；闭关可为后续功法感悟预留接口，但本轮不落地。
- 不实现突破材料消耗链；若后续突破消耗材料，失败时材料全部浪费，但材料本身不在本轮范围。
- 不实现天劫、心魔、境界跌落、突破失败死亡。
- 不重做功法、法宝、丹药和 GAS 效果体系。

## 状态字段

运行时只保留以下修为相关字段：

| 字段 | 含义 |
|------|------|
| `cultivation` | 闭关、修炼场、丹药等直接修炼获得的修炼修为 |
| `experienceCultivation` | 游历、任务、斩妖、动态事件、机会点、PvP、外出社交等获得的历练修为 |
| `totalCultivation` | `cultivation + experienceCultivation`，可作为派生缓存或即时计算结果 |
| `qi` | 真气，突破硬门槛之一 |
| `rankStage` | 境界小层：`early`、`middle`、`late`、`perfection`，凡人为 `null` 或按现有战斗属性兼容规则处理 |

以下字段必须从业务语义中删除：

```text
cultivationProgress
insight
totalProgress
cultivationProgressRatio
```

删除范围包括运行时状态初始化、GOAP 状态、行为配置、快照输出、报表、跟随面板、测试断言和文档主语义。不得以“短期兼容字段”的形式继续输出这些比例字段。

## 境界需求与小层

`ranks.json` 继续保存每个目标境界的 `qiRequired` 与 `cultivationRequired`。本轮要求：

```text
cultivationRequired === qiRequired
```

保留 `cultivationRequired` 字段的原因是 UI、报表和平衡配置可直接读取“突破所需修为”；后续如果修为与真气门槛分离，不需要再调整数据结构。

小层由当前目标境界的修为完成度推导：

```text
completion = totalCultivation / cultivationRequired
```

| 完成度 | 小层 |
|--------|------|
| `0 <= completion < 0.25` | 初期 `early` |
| `0.25 <= completion < 0.60` | 中期 `middle` |
| `0.60 <= completion < 0.90` | 后期 `late` |
| `completion >= 0.90` | 圆满 `perfection` |

小层推进不触发突破失败判定。只有大境界圆满并满足突破门槛时，才进入正式突破。

## 统一接口

`apps/game/js/engine/npc/numeric-cultivation.js` 升级为修为规则唯一入口。其他模块不直接计算旧比例，不直接拼接突破字段结算。

建议接口：

```text
getNextCultivationRank(entity, ranks)
getCultivationRequired(entity, ranks)
getTotalCultivation(entity)
syncTotalCultivation(entity)
computeRankStage(entity, ranks, thresholds)
refreshRankStage(entity, ranks)

addCultivation(entity, ranks, amount)
addExperienceCultivation(entity, ranks, amount)

computeCultivationGain(entity, ranks, baseGain, config)
canAttemptBreakthrough(entity, ranks, config)

applyBreakthroughSuccess(entity, nextRank, config)
applyBreakthroughFailure(entity, nextRank, config)
```

调用边界：

| 模块 | 职责 |
|------|------|
| `npc-state.js` | 初始化数值修为字段，不再同步比例字段 |
| `cultivation-service.js` | 负责闭关收益、真气收益、灵石消耗，调用 `addCultivation()` |
| `cultivation-experience.js` | 负责历练收益公式，调用 `addExperienceCultivation()` |
| `npc-lifecycle.js` | 负责触发突破，成功/失败字段结算调用统一接口 |
| UI、报告、快照 | 显示修为数值、小层和突破需求，不再显示旧比例字段 |

## 闭关修为

旧 `cultivationCap` 硬上限移除。闭关永远可以增加 `cultivation`，但收益随闭关修为完成度递减：

```text
completion = cultivation / cultivationRequired
decay = exp(-cultivationDecayK * completion)
cultivationGain = baseGain * decay
```

设计含义：

- 刚进入新境界时闭关收益正常。
- 闭关修为越高，继续闭关越慢。
- 历练修为不会让闭关变慢。
- 纯闭关理论上可以攒满突破修为，但耗时很长。
- 闭关后续可作为功法感悟来源，但功法感悟不在本轮实现。

`cultivationDecayK` 可继续沿用现有配置名，但注释必须改为“闭关修为递减系数”，不再描述为接近 cap 的递减。

## 历练修为

历练、任务、斩妖、动态事件、机会点、PvP、外出社交等继续通过统一入口增加 `experienceCultivation`：

```text
addExperienceCultivation(entity, ranks, gain)
```

历练收益仍按事件价值、风险、耗时、结果倍率计算。高价值、高风险事件给更多修为，但必须伴随真实风险：受伤、死亡、任务失败、妖兽袭击、敌对关系或资源损失等。

历练修为不影响闭关递减，避免“历练多了回来闭关反而变慢”的不直观结果。

## 突破判定

正式突破只发生在大境界之间。判定条件为：

```text
totalCultivation >= cultivationRequired
cultivation >= cultivationRequired * minCultivationRatio
qi >= qiRequired
```

`minCultivationRatio` 暂时沿用 `0.3`，但语义改为“最低闭关修为占比”，不是旧进度条比例。它保证角色至少有一部分闭关根基，不能完全依靠历练修为跳境界。

## 突破成功

突破成功后：

```text
rankId = nextRank.id
rankName = nextRank.name
rankStage = early
cultivation = 0
experienceCultivation = 0
totalCultivation = 0
qi -= qiRequired
```

同时刷新寿元、战斗属性、闭关/历练偏好和相关行为状态。普通突破不默认回满血；突破回满血仍属于特殊功法、体质或 GAS 效果。

不保留溢出修为，避免连续突破，也符合“本境界积累转化为新境界根基”的语义。

## 突破失败

突破失败后：

```text
qi = floor(qi * 0.5)
cultivation = cultivation * 0.2
experienceCultivation = experienceCultivation * 0.2
totalCultivation = cultivation + experienceCultivation
injuryLevel = max(currentInjuryLevel, 3)
```

含义：

- 真气是冲关燃料，失败后损失 50%。
- 修为伤及根基，只保留 20%。
- 历练机缘也随冲关失败大幅散失，本轮与闭关修为使用同一保留比例。
- 失败必定导致重伤，但不引入境界跌落、天劫死亡或心魔。
- 若未来突破消耗材料，失败时已消耗材料全部浪费；材料消耗链另行设计。

配置建议：

```json
{
  "breakthrough": {
    "failureQiRetention": 0.5,
    "failureCultivationRetention": 0.2,
    "failureInjuryLevel": 3
  }
}
```

## AI 行为选择

取消闭关硬上限后，AI 必须能看到闭关收益递减，否则可能重新退化为长期闭关。

行为评估应同时包含理性层和性格/执念层：

```text
行为效用 = 预期收益 / 时间成本
        - 风险惩罚
        + 性格/执念偏好
        + 当前短板修正
```

理性层：

- 闭关预期收益读取递减后的 `cultivationGain`。
- 历练预期收益读取事件价值、风险、耗时和结果倍率。
- 真气不足时，闭关、丹药、灵石和被动吸纳相关行为权重提高。
- 修为不足时，闭关和历练都可成为候选，但闭关收益会随 `cultivation / cultivationRequired` 下降。

性格/执念层：

| 倾向 | 行为偏好 |
|------|----------|
| 谨慎、怕死 | 闭关、低风险任务、修炼场、疗伤 |
| 勇敢、好胜 | 历练、斩妖、挑战、动态事件 |
| 贪婪 | 高价值高风险任务、秘境、夺宝 |
| 长生执念 | 求丹、稳妥机缘、修炼场、低风险成长 |
| 宗门责任 | 宗门任务、动态事件、护法、带徒 |

本轮不要求重做所有 Utility 公式，但必须让决策层能读取闭关递减收益，避免只在执行层变慢而规划层无感。

## 验证

Focused tests：

- `NPCState` 不再初始化旧比例字段。
- GOAP 状态、快照、UI 主逻辑不再输出旧比例字段。
- `addCultivation()`、`addExperienceCultivation()` 正确累加并刷新 `totalCultivation`。
- `rankStage` 按 `totalCultivation / cultivationRequired` 阈值刷新。
- 闭关收益随 `cultivation / cultivationRequired` 下降。
- 历练修为不影响闭关递减。
- 突破成功清零两类修为、扣除真气、进入下一境界初期。
- 突破失败真气保留 50%、两类修为保留 20%、`injuryLevel` 至少为 3。

行为验证：

- AI 能读取递减后的闭关收益。
- 闭关收益很低时，历练、任务、斩妖等候选行为的效用能自然抬升。
- 谨慎和勇敢角色在相同修为状态下能出现不同选择倾向。

真实模拟：

- 多种子跑完整世界，观察突破次数、突破失败重伤、闭关/历练/任务分布。
- 检查 NPC 不再卡在旧 `progress` 字段或旧 cap 前置条件。
- 检查历练高收益伴随真实风险，而不是无脑升级。
- 检查真气与修为共同成为突破约束，不互相污染；`yuan/maxYuan` 仍只属于战斗资源。

验证不得使用哈希、指纹或单一摘要值自证；必须观察真实行为数据。
