# 妖兽伤害、斩妖任务与历练修为规格

> 最后更新：2026-06-05
> 状态：待实施规格
> 关联文档：`docs/decisions/adr-011-cultivation-incentive-system.md`、`docs/decisions/adr-017-value-risk-decision-and-cultivation-curve.md`、`docs/decisions/adr-039-qi-progress-decoupling.md`、`docs/decisions/adr-047-cultivation-action-selection-balance.md`、`docs/systems/job-toil-ai-spec.md`、`docs/data/data-config-rules.md`
> 关联代码与数据：`apps/game/js/engine/monster/monster-entity.js`、`apps/game/data/balance/monster-spawn.json`、`apps/game/js/engine/monster/monster-resources.js`、`apps/game/js/engine/npc/actions/quest-actions.js`、`apps/game/js/engine/npc/actions/npc-action-utils.js`、`apps/game/data/quests/quest-templates.json`、`apps/game/data/balance/cultivation.json`、`apps/game/data/definitions/ranks.json`
> 来源：用户 2026-06-05 对“调低妖兽主动攻击伤害、让斩妖任务真实杀怪、非闭关事件给历练修为、修炼进度改为明确数值修为”的要求；用户随后明确修订：旧 NPC Action 需先迁移为 Job/Toil，新行为必须按新系统添加配置和代码，战斗智能与统一战斗公式也纳入本次范围。

## 背景

当前炼气及以下人物在地图妖兽主动袭击中死亡过快。`MonsterEntity._attack()` 的单击伤害为：

```text
damage = monster.power * (1 - npcDef) * random(0.8, 1.2)
```

`monster-spawn.json` 中的 `killChanceFactor` 影响猎杀倾向和击杀概率语义，但没有进入这条真实扣血公式。NPC 反击妖兽仍是 `npcCounterDamageBase + npcOrder * npcCounterOrderWeight + random(0..10)`，斩妖任务胜率走 `npcCombatPower` 与 `monsterCombatPower`，两者都不应被本次伤害倍率误伤。

斩妖任务已有半套真实链路：接取任务时会写入 `questTargetX`、`questTargetY`、`questTargetMonsterId`，执行任务末尾会调用 `settleMonsterHunt()`，成功时把地图上的具体妖兽标记为 `quest_hunt` 死亡并发放掉落。但任务实例仍偏扁平，缺少明确的任务类型、价值、风险、妖兽名、数量等结构化字段，也没有把“外出历练带来修为增长”做成统一规则。

修炼系统当前以 `cultivationProgress` 与 `insight` 的 0..1 比例相加为 `totalProgress`，突破再检查 `qi`。用户要求把“修炼进度”改为“修为”，并像真气一样显示和结算为明确数值。

## 目标

1. 先把旧 NPC Action 全量迁移到 Job/Toil：任务、游历、战斗、修炼收益、社交外出等 NPC 行为不能继续在旧 Action executor 上扩展。
2. 新增行为必须按新系统添加配置和代码：`npc-job-actions.json`、`apps/game/data/jobs/`、`apps/game/data/toils/` 与对应 Toil executor 是主路径。
3. 建立统一战斗/风险结算管线，让地图妖兽主动攻击、斩妖任务、PvP、普通任务风险都进入同一套可配置战斗公式和结果结构。
4. 在统一公式中降低地图妖兽主动攻击 NPC 的场景伤害倍率，缓解低阶人物被一阶妖兽过快击倒。
5. 把斩妖、除害、猎灵兽这类杀怪任务做成类型化 Job 任务实例，必须记录坐标、价值、风险、目标妖兽名、目标妖兽 id、目标阶位、数量与击杀进度。
6. NPC 接到杀怪任务后必须真的出门、移动到任务目标、评估风险、准备补给，必要时撤退、疗伤、避开明显高阶目标或尝试组队。
7. 任务、动态事件、游历、机会点、PvP、外出社交等非闭关、非原地待命事件都通过统一入口获得额外历练修为；历练修为按事件价值和风险放大。
8. 把 `cultivationProgress`/`insight` 的比例语义迁移为明确数值修为：`cultivation`、`experienceCultivation`、`totalCultivation`，并保留突破所需的闭关修为最低占比和外出历练补足机制。

## 当前范围

- 重做人物战斗智能的最低可用闭环：危险评估、补给准备、撤退疗伤、避开高阶妖兽、可选组队。
- 统一地图妖兽主动攻击、斩妖任务、PvP、普通任务风险的战斗/风险结算公式，旧的分散胜率与风险扣血只能作为迁移输入，不再作为新增行为的主路径。
- 旧 Action 立刻迁移到 Job/Toil。旧 executor 可以作为迁移参考或测试夹具，但本规格实施后，新增杀怪、历练、战斗智能行为不得继续落在旧 Action 链路。
- 验证必须看真实模拟现象；不用摘要一致或关闭妖兽生成来证明正确。

## 方案选择

### 方案 A：只调低妖兽主动伤害

优点是改动小、能最快缓解暴毙。缺点是继续保留旧 Action 和分散战斗公式，后续杀怪任务、历练修为和战斗智能还会继续绕过 Job/Toil。

### 方案 B：Job/Toil 优先重构，再落地杀怪、战斗智能和数值修为

优点是先把旧 Action 迁到新系统，之后所有新行为都按 Job/Toil 配置和 Toil executor 增长，架构方向一致；统一战斗公式、杀怪任务、历练修为和数值修为都能在同一套数据流里验证。缺点是第一阶段实施量变大，必须补更多迁移测试。

### 方案 C：直接重写全部 NPC AI

优点是最终形态更完整。缺点是会把 Utility、GOAP、Job/Toil、战斗、任务、修为全部推倒重来，超出本次目标，也更难与现有 ADR-050 架构衔接。

本规格采用方案 B。

## 阶段一：旧 NPC Action 全量迁移到 Job/Toil

### 迁移原则

本次实施前置条件是：NPC 行为层旧 Action 不再作为真实执行主路径。`apps/game/data/actions/npc-actions.json` 中已有 NPC 行为需要迁移成 JobAction + Job + Toil：

- 修炼链：闭关、修炼场、兑换/服用丹药、疗伤。
- 任务链：接取任务、执行任务、交付任务、上交材料。
- 游历链：游历历练、机会点收益、动态事件参与。
- 战斗链：复仇追踪、PvP 击杀、杀怪任务、妖兽遭遇。
- 社交外出链：师徒点化、拜访、结伴请求。

迁移后的结构：

```text
npc-job-actions.json      只描述 GOAP 可规划的 JobAction 与 plannerEffects
apps/game/data/jobs/      描述 Job 编排、输入、成功效果、打断策略
apps/game/data/toils/     描述 Toil 类型、参数、目标解析器
js/engine/npc/toils/      放真实执行代码
```

旧 Action executor 可以临时保留给测试或回退分支，但普通世界模拟、默认 NPC 行为集和新增行为不得再调用旧 executor。实施计划必须包含“旧 Action 不再被默认 action set 使用”的配置加载测试。

### 旧 Action 删除规则

迁移完成后可以删除的是“直接执行型 NPC Action”及其无引用代码：

- `apps/game/data/actions/npc-actions.json` 中已经迁移为 JobAction 的旧 NPC Action 配置。
- `apps/game/data/actions/npc-action-sets.json` 中对这些旧 NPC Action id 的默认引用。
- `apps/game/js/engine/npc/actions/` 下只服务这些旧 NPC Action、且无任何测试或兼容入口引用的 executor。
- `apps/game/js/engine/npc/npc-actions.js` 中对无引用旧 executor 的 import 与 `ActionPool.registerExecutor()` 注册。

不能删除的是 Action/GOAP 抽象层本身：

- `apps/game/js/engine/abstract/action.js`、`goap-planner.js`、`behavior-system.js` 等规划抽象仍保留。
- JobAction 仍然是 GOAP 可规划的 Action，只是 `executionKind:"job"`，真实执行下沉到 Job/Toil。
- `npc-job-actions.json` 是迁移后的 Action 配置主入口，不能因为删除旧 `npc-actions.json` 条目而删除 JobAction。

删除前必须通过无引用验证：

1. 默认 NPC action set 中不存在已迁移旧 Action id。
2. `npc-actions.json` 中不存在已迁移旧 Action id。
3. `npc-job-actions.json` 中存在对应 JobAction。
4. JobAction 引用的 Job、Job 引用的 Toil、Toil executor 都能加载。
5. 真实模拟中旧 executor 执行次数为 0，JobAction、Job、Toil 执行次数大于 0。
6. `rg` 或配置加载测试确认被删 executor 没有生产代码引用；若只剩测试引用，测试应迁移到 Job/Toil 路径后再删。

### Job/Toil 准入规则

新增行为必须同时具备：

1. 一个 `executionKind:"job"` 的 JobAction，写入 `apps/game/data/actions/npc-job-actions.json`。
2. 一个或多个 Job 配置，写入 `apps/game/data/jobs/*.json`。
3. 必要 Toil 配置，写入 `apps/game/data/toils/*.json`。
4. 必要 Toil executor，写入 `apps/game/js/engine/npc/toils/`。
5. 配置加载测试，断言 JobAction 引用的 Job、Job 引用的 Toil、Toil 引用的物品/目标解析器都存在。

如果某个行为暂时无法迁移，必须在实施计划中单独列为阻塞项，不能在旧 Action 上继续扩展同名新行为。

## 阶段二：统一战斗公式与妖兽主动伤害倍率

### 数据规则

在 `apps/game/data/balance/monster-spawn.json` 的顶层 `combat` 增加：

```json
"damageMultiplier": 0.75
```

注释必须说明该倍率是统一战斗公式中的 `monster_ambush` 场景参数，只影响“地图妖兽主动攻击 NPC”的场景伤害系数。NPC 反击、斩妖任务、PvP、普通任务风险也进入统一战斗/风险结算管线，但使用各自的场景系数，不读取这个倍率：

- NPC 反击妖兽场景。
- 斩妖任务场景。
- PvP 场景。
- `risk.json` 的普通任务、游历、夺宝等风险场景。

`behaviorByTier.tier1/tier2/tier3` 可以继承顶层默认值。若未来需要分档调参，再在 tier 配置里覆盖 `damageMultiplier`。

### 统一战斗输入

新增统一战斗/风险结算服务，建议位置：

```text
apps/game/js/engine/combat/combat-encounter.js
```

接口语义：

```js
resolveCombatEncounter({
  attacker,
  defender,
  scene,
  power,
  defense,
  value,
  riskScore,
  random,
  worldContext,
})
```

首批 `scene`：

| scene | 使用场景 |
| --- | --- |
| `monster_ambush` | 地图妖兽主动攻击 NPC。 |
| `monster_counter` | NPC 被妖兽袭击后的反击。 |
| `monster_hunt_quest` | 斩妖、除害、猎灵兽任务。 |
| `pvp` | NPC 之间真实交战。 |
| `quest_risk` | 普通任务中的战斗/危险遭遇。 |

统一结果结构必须包含：`hit`、`damage`、`winChance`、`died`、`injuryGain`、`retreatSuggested`、`deathInfo`、`experienceContext`。旧 `settleMonsterHunt()`、`MonsterEntity._attack()`、PvP 和任务风险结算都迁移为调用该服务。

### 代码规则

`MonsterEntity._attack()` 的伤害公式改为：

```text
damage = monster.power * (1 - npcDef) * random(0.8, 1.2) * damageMultiplier
```

读取顺序：

1. 优先读当前妖兽实例 `_combat.damageMultiplier`。
2. 其次读 `monster-spawn.json.combat.damageMultiplier` 注入后的默认值。
3. 缺省为 `1.0`，保证旧数据可运行。

`MonsterEntity._attack()` 不再直接拥有完整战斗公式；它只收集攻击者、受击者、场景和倍率，然后调用 `resolveCombatEncounter(scene:"monster_ambush")`。同理，斩妖任务和 PvP 不再各自维护独立胜率公式，而是通过不同 `scene`、不同系数和不同战力输入进入同一服务。

### 分析脚本

`apps/game/tools/monster-vs-npc.mjs` 必须读取同一个 `damageMultiplier`，并输出倍率生效后的凡人、炼气、筑基受击对比。输出重点：

- 一阶妖兽面对凡人/炼气不再普遍 1-2 击立即倒。
- 二阶妖兽仍危险。
- 高阶妖兽仍能碾压低阶人物。

## 阶段三：Job/Toil 类型化杀怪任务实例

### 任务实例模型

接取任务后，NPC 身上必须形成一个结构化任务实例。该实例由 Job/Toil 产生和推进，旧的扁平 state 字段只允许作为 UI/兼容派生字段，单一真相源应收敛到下列语义：

```json
{
  "id": "quest_npc_001_day_120",
  "templateId": "qt_slay_monster",
  "type": "monster_hunt",
  "name": "二阶斩妖",
  "category": "combat",
  "difficulty": 2,
  "value": 10,
  "riskKey": "monster_hunt",
  "riskScore": 0.055,
  "source": "quest_hall",
  "state": "accepted",
  "target": {
    "kind": "monster",
    "x": 123,
    "y": 45,
    "monsterIds": ["monster_abc"],
    "monsterName": "青鳞狼",
    "monsterGrade": 2,
    "requiredKills": 1,
    "killedCount": 0
  },
  "rewards": {
    "stones": 10,
    "contribution": 5,
    "factionStones": 25,
    "rewardProfileId": "qt_slay_monster"
  }
}
```

字段含义：

| 字段 | 规则 |
| --- | --- |
| `type` | 杀怪任务统一为 `monster_hunt`；普通巡山、采药、护送等继续使用各自类型。 |
| `value` | 来自难度奖励、任务类型价值和目标妖兽资源价值；用于 GOAP 价值估算与历练修为加成。 |
| `riskScore` | 来自难度 `dangerInjury + dangerDeath`、目标妖兽阶位和战力差；用于风险决策与历练修为加成。 |
| `target.x/y` | 接取任务时锁定的地图坐标；执行任务必须先移动到这里或目标妖兽的当前位置。 |
| `target.monsterIds` | 目标妖兽实例 id 列表；数量大于 1 时逐个击杀或重定向到同阶附近妖兽。 |
| `target.monsterName` | 接取任务时记录的妖兽名，用于日志、UI、验证和死亡原因。 |
| `target.requiredKills` | 任务要求数量；首批斩妖/除害为 1，猎灵兽可按难度配置为 1-3。 |
| `target.killedCount` | 已真实击杀数量；只有地图活体妖兽被标记死亡后才增加。 |

### 接取规则

`toil_accept_quest` 或 `toil_bind_monster_hunt_quest` 选择 `locationTarget:"monster"` 的任务时必须：

1. 从地图活体妖兽中选择与任务难度匹配的目标。
2. 写入目标坐标、妖兽 id、妖兽名、阶位、数量、价值、风险。
3. 如果没有可用妖兽，不能生成“空目标杀怪任务”；该候选任务应跳过或返回无可接取任务。
4. 旧字段 `questTargetX`、`questTargetY`、`questTargetMonsterId` 可以由任务实例派生，不能成为互相矛盾的第二份真相源。

### 执行规则

`job_npc_monster_hunt` 执行杀怪任务时必须满足：

1. `toil_move_to_target` 使用 `quest_target` 或 `monster_hunt_target` 解析器，NPC 不到目标地不进入击杀结算。
2. `toil_assess_monster_hunt_risk` 必须比较 NPC 战力、补给、伤势、目标妖兽阶位和距离；明显不可打时转入撤退、换目标、请求同伴或放弃任务。
3. `toil_prepare_monster_hunt` 必须按风险准备疗伤丹、符箓、基础法器等补给；补给不足时先走经济/兑换 Job，或降低可接任务难度。
4. 目标妖兽仍活着时，只能结算该妖兽。
5. 目标妖兽已被别人杀死或迁移失联时，允许按同阶位、同任务坐标附近重新锁定目标；重定向必须写回新的妖兽 id、名称、坐标。
6. 找不到替代目标时任务失败为 `target_lost`，不能凭空完成。
7. `toil_hunt_monster_target` 必须调用统一 `resolveCombatEncounter(scene:"monster_hunt_quest")`，成功时让被杀妖兽 `alive=false`、`_deathInfo.cause='quest_hunt'`，并进入 `tickLog.monsterDeaths`。
8. `killedCount` 达到 `requiredKills` 后才置 `questComplete=true`。

首批新增配置：

### 任务日志

任务日志和 `infoEvents` 至少包含：

- NPC 名称与 id。
- 任务类型、难度、价值、风险。
- 目标妖兽名、妖兽 id、阶位。
- 目标坐标。
- 击杀数量进度。
- 成功、失败、目标失踪、NPC 死亡或受伤结果。

## 阶段四：NPC 战斗智能最低闭环

本阶段不是完整战术 AI，但必须让 NPC 在出门杀怪和遭遇妖兽时具备基本生存判断，避免“明知必死还照旧执行任务”。

### 状态与评估

NPC Job 状态需要暴露：

| 状态 | 规则 |
| --- | --- |
| `combatRiskScore` | 由目标阶位、战力差、伤势、补给、距离、目标数量计算。 |
| `combatReady` | 有最低补给、伤势未过重、目标不明显碾压时为 true。 |
| `shouldRetreat` | 当前 HP、伤势或战力差触发撤退阈值。 |
| `needsCombatSupply` | 缺疗伤丹、符箓、法器或真气不足。 |
| `needsCompanion` | 风险高但收益高，且附近有同门/关系目标可邀请。 |
| `monsterTooDangerous` | 目标阶位高出自身可承受范围。 |

### 行为要求

新增或迁移下列 Job/Toil：

```text
job_npc_prepare_combat
job_npc_retreat_and_heal
job_npc_request_hunt_companion
toil_assess_combat_risk
toil_prepare_combat_supply
toil_request_companion
toil_retreat_to_safe_place
toil_use_heal_item
toil_abort_overdangerous_target
```

规则：

1. 杀怪任务执行前必须先过 `toil_assess_combat_risk`。
2. `monsterTooDangerous=true` 时不能继续硬杀；必须换目标、请求同伴或放弃任务。
3. `needsCombatSupply=true` 时先进入补给 Job，补给失败则降低任务难度或放弃。
4. 战斗中 `shouldRetreat=true` 时进入撤退疗伤 Job，不得继续执行击杀 Toil。
5. 高价值高风险任务允许触发 `job_npc_request_hunt_companion`，但组队失败不能卡死任务链。

## 阶段五：历练修为统一入口

### 新增统一函数

新增共享结算函数，建议位置：

```text
apps/game/js/engine/npc/cultivation-experience.js
```

接口语义：

```js
applyCultivationExperience(entity, worldContext, {
  sourceKind,
  value,
  riskScore,
  durationDays,
  outcome,
  description,
})
```

`sourceKind` 首批取值：

| sourceKind | 触发点 |
| --- | --- |
| `quest_progress` | 普通任务或杀怪任务推进中的每日外出执行。 |
| `quest_complete` | 普通任务完成。 |
| `monster_hunt_success` | 杀怪任务真实击杀妖兽。 |
| `monster_hunt_failure` | 杀怪任务失败但 NPC 未死。 |
| `explore` | `job_npc_explore` 游历归来。 |
| `dynamic_event` | 动态事件参与成功。 |
| `opportunity` | 机会点收益结算。 |
| `pvp` | 真实 PvP 交战后存活。 |
| `social_travel` | 师徒点化、拜访等外出社交行为完成。 |

不触发历练修为的行为：

- `job_npc_cultivate` 闭关修炼。
- 原地待命、等待事件阶段、纯 UI 或纯状态刷新。
- 已死亡 NPC 的失败结果。

### 增益公式

首批公式写入 `cultivation.json.experience`：

```json
{
  "enabled": true,
  "valueScale": 500,
  "riskWeight": 0.75,
  "maxValueMultiplier": 3,
  "maxRiskMultiplier": 3,
  "maxDurationMultiplier": 2.5,
  "baseBySource": {
    "quest_progress": 0.35,
    "quest_complete": 2,
    "monster_hunt_success": 8,
    "monster_hunt_failure": 2,
    "explore": 5,
    "dynamic_event": 6,
    "opportunity": 5,
    "pvp": 8,
    "social_travel": 1
  },
  "outcomeMultiplier": {
    "success": 1,
    "partial": 0.5,
    "failure": 0.35
  }
}
```

计算规则：

```text
valueMultiplier = min(maxValueMultiplier, 1 + log1p(max(0, value)) / log1p(valueScale))
riskMultiplier = min(maxRiskMultiplier, 1 + max(0, riskScore) * riskWeight)
durationMultiplier = min(maxDurationMultiplier, 1 + log1p(max(1, durationDays)) / log1p(90))
gain = baseBySource[sourceKind] * valueMultiplier * riskMultiplier * durationMultiplier * outcomeMultiplier[outcome]
```

结果写入 `experienceCultivation`，再同步 `totalCultivation`。日志显示为 `历练修为+X.X`。

### 与现有游历的关系

`act_npc_explore` 当前给 `insight` 与 `qi`。迁移后：

- `insight` 改为 `experienceCultivation`。
- 游历仍可给真气 `qi`。
- 原本 `fortuneEvents[].insightMultiplier` 改为修为倍率，语义为“历练修为倍率”。
- 外出任务、动态事件和机会点不再绕过修为系统，全部走同一入口。

`job_npc_explore`、`job_npc_monster_hunt`、`job_npc_join_dynamic_event`、`job_npc_visit_relationship_target` 等 Job 完成时，都必须通过 Toil 或 Job success hook 调用 `applyCultivationExperience()`。旧 Action executor 中的 `insight` 增加逻辑迁移后不得再作为真实路径存在。

## 阶段六：修炼进度迁移为数值修为

### 状态字段

NPC state 新增并逐步替换旧字段：

| 新字段 | 语义 |
| --- | --- |
| `cultivation` | 闭关、修炼场、丹药等直接修炼获得的数值修为。 |
| `experienceCultivation` | 外出历练、任务、事件、战斗等获得的数值修为。 |
| `totalCultivation` | `cultivation + experienceCultivation`，突破修为门槛使用。 |
| `cultivationProgressRatio` | `totalCultivation / nextCultivationRequired`，仅用于兼容旧 GOAP 或百分比 UI，不作为显示主语义。 |

旧字段迁移：

| 旧字段 | 迁移规则 |
| --- | --- |
| `cultivationProgress` | `cultivation = cultivationProgress * nextCultivationRequired` |
| `insight` | `experienceCultivation = insight * nextCultivationRequired` |
| `totalProgress` | 改为派生兼容字段：`totalCultivation / nextCultivationRequired` |

### 境界需求数值

在 `apps/game/data/definitions/ranks.json` 的修仙境界上补 `cultivationRequired`，数值与 `qiRequired` 同一数量级但独立：

| 目标境界 | `qiRequired` | `cultivationRequired` |
| --- | ---: | ---: |
| 炼气 | 50 | 100 |
| 筑基 | 500 | 1000 |
| 金丹 | 5000 | 10000 |
| 元婴 | 50000 | 100000 |
| 化神 | 500000 | 1000000 |

突破检查改为：

```text
totalCultivation >= nextRank.cultivationRequired
qi >= nextRank.qiRequired
cultivation >= nextRank.cultivationRequired * minCultivationRatio
```

闭关上限从比例换算为数值：

```text
retreatCultivationCap = nextRank.cultivationRequired * cultivationCapRatio[currentRank]
```

其中 `cultivationCap` 可以在第一轮实现中保留原字段名，但文档和注释必须说明它是“闭关修为上限比例”，不是最终显示给玩家的修为数值。

### 修炼收益换算

为尽量保持节奏，旧的 `cultivationSpeed[rank]` 先按比例转成数值收益：

```text
cultivationGainPerDay = cultivationSpeed[rank] * nextRank.cultivationRequired
```

原 `qiPerProgress` 改为按修为产气：

```text
qiGainFromCultivation = qiPerCultivation[rank] * actualCultivationGain
qiPerCultivation[rank] = qiPerProgress[rank] / nextRank.cultivationRequired
```

丹药、灵果、强者精血、GE `ge_add_progress` 需要迁移为数值修为效果。新通用 GE 建议命名为 `ge_add_cultivation`，具体数值继续由物品 `effects[].magnitude` 提供，遵守 ADR-042 的“GE 是通用机制原语”规则。

### UI 和文案

- “闭关进度”改为“修为”或“闭关修为”。
- “游历感悟”改为“历练修为”。
- “突破总进度”改为“总修为/突破所需修为”。
- UI 可以保留百分比条，但必须同时显示明确数值，例如 `总修为 642 / 1000`，真气显示仍为 `真气 530 / 500`。

## 验收标准

1. 一阶妖兽仍有威胁，但炼气人物不再普遍被一阶妖兽两击内带走。
2. 二阶及以上妖兽对低阶人物仍危险，高阶妖兽仍能形成世界风险。
3. 默认 NPC 行为主路径已经迁到 Job/Toil，新行为没有落在旧 Action executor。
4. 地图妖兽主动攻击、NPC 反击、斩妖任务、PvP、普通任务风险都走统一战斗/风险结算服务。
5. 杀怪任务没有“抽象完成”：完成日志必须对应至少一个地图活体妖兽真实死亡。
6. 杀怪任务实例包含任务类型、坐标、价值、风险、妖兽名、数量和击杀进度。
7. NPC 在明显危险时能撤退、疗伤、补给、避开高阶目标或尝试组队，不能无脑硬打。
8. 非闭关、非原地待命事件能通过统一入口增加历练修为，且高价值、高风险事件给更多修为。
9. 修为不再只是 0..1 进度条；NPC 状态、UI 和突破判定都能看到明确数值修为。
10. 所有验证基于真实配置和真实模拟，不用摘要一致代替行为观察，不关闭妖兽生成，不给特定 NPC 开特权。

## 自检

- 本规格已把旧 NPC Action 迁移、统一战斗公式、战斗智能、杀怪任务、历练修为和数值修为纳入当前范围。
- 杀怪任务明确要求通过 Job/Toil 让真实地图妖兽死亡，不再停留在胜率文本。
- 历练收益是共享入口，不把同一公式散落到多个执行器。
- 数值修为保留旧节奏换算方法，并明确旧字段迁移与兼容策略。
- 未使用摘要一致或关闭系统作为验收依据。
