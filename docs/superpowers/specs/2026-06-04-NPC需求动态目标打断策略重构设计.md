# NPC 需求-动态目标-打断策略重构设计

> 最后更新：2026-06-04
> 状态：设计稿，待用户复核后进入实施计划。
> 来源：用户在 2026-06-04 对 NPC AI 架构优化的讨论；现有实现参考 `docs/decisions/adr-048-four-layer-reactive-ai.md`、`apps/game/data/behavior-trees/npc-default.json`、`apps/game/js/engine/abstract/bt/reactions.js`、`apps/game/js/engine/npc/intent-service.js`、`apps/game/js/engine/npc/npc-goals.js`、`apps/game/js/engine/npc/npc-utility.js`、`apps/game/js/engine/abstract/behavior-system.js`。

## 1. 背景

当前 NPC AI 已形成四层结构：Reaction 即时反应、Utility 意图选择、GOAP 短链规划、Execution 执行推进。这个结构能表达被攻击后的逃命、回血、撤退和反击，也能通过 `requestReplan()` 支持大事件立即重决策。

现有问题在于“需求”和“行为/流程目标”的边界不够清晰。`npc-needs.json` 中既有修炼、回血、职责这类长期动机，也有完成当前任务、上交材料、猎取妖兽资源这类具体工作流目标。随着秘境、宗门大比、天材地宝、高手陨落、亲人被杀等事件增多，如果继续把事件目标直接塞进基础需求，会让需求表越来越像任务表，NPC 的人格也会变得不清楚。

本设计目标是重构边界，让 NPC 的行为从“被事项推着走”变成“由人生动机、世界事件和处境共同驱动”。

## 2. 设计目标

1. 基础需求只表达长期人生动机，不再承载具体流程。
2. 临时事件通过动态目标接入，不污染常驻需求。
3. 未来事件需要能触发提前准备，例如秘境 90 天后开启时，NPC 可以出关备药、攒灵石、召集同门、提前赶路。
4. 打断闭关和当前行为必须有明确策略，而不是“有事件就打断”或“永远不打断”。
5. 行为层保持原子化或短流程化，同一个行为可被不同动机复用。
6. 保持与现有四层 AI 兼容，优先演进 `IntentService`、`npc-goals.js`、`npc-utility.js`、`PlannerNode` 接缝，不重写整套引擎。

## 3. 总体架构

新的 NPC 意图模型分为五个概念层：

```text
长期需求 Need
  ↓
世界事件 WorldEvent / 消息感知 EventAwareness
  ↓
动态目标 DynamicGoal
  ↓
打断策略 InterruptPolicy
  ↓
GOAP 行为计划与 Execution 执行
```

四层 AI 的对应关系：

```text
Reaction 层
  处理生死级即时刺激：被攻击、濒死、贴脸威胁。

Utility / Intent 层
  合并长期需求、执念、关系目标、机会目标、动态目标。

GOAP 层
  只回答如何达成目标。

Execution 层
  推进移动、执行、结算，不表达为什么做。
```

## 4. 需求层

基础需求应收敛为长期动机：

| 动机 | 含义 | 示例目标来源 |
| --- | --- | --- |
| 求道 | 境界、突破、修炼成长 | 缺真气、缺进度、缺破境助力 |
| 求生 | 活下去、疗伤、延寿、避险 | 回血、逃离强敌、寻找续命资源 |
| 求利 | 资源、丹药、法宝、灵石、机缘 | 夺宝、秘境、高手遗产、拍卖 |
| 求权 | 职位、掌门、势力话语权 | 晋升、挑战、夺权 |
| 求情 | 师徒、道侣、亲族、同门、恩义 | 护徒、报恩、奔丧、驰援 |
| 求名/责 | 宗门职责、名望、护道 | 月度贡献、大比、宗门危机 |
| 求仇 | 复仇、追杀、清算 | 亲人被杀、师傅被杀、宿敌现身 |
| 求安 | 养老、归隐、避世 | 回洞府、减少风险、传承安排 |

我的判断：旧的“完成当前任务、上交材料、猎妖资源”不应作为核心需求存在，而应成为目标解析后的阶段目标或 GOAP 子目标。例如“上交材料”可能由求道、求责、求利三种动机触发，行为本身不应绑定唯一需求。

## 5. 世界事件与感知

事件只描述世界发生了什么，不直接决定 NPC 做什么。

核心事件类型：

| 事件类型 | 示例 |
| --- | --- |
| 秘境预告/开启 | 90 天后秘境开启、秘境已开启 |
| 宗门大比预告/开启 | 30 天后大比、今日报名 |
| 拍卖会预告/开启 | 坊市放出拍品消息 |
| 天材地宝出世 | 灵草成熟、法宝现世 |
| 高手陨落 | 高手死亡，其遗产可能可夺 |
| 关系伤亡 | 道侣、亲人、师傅、徒弟、同门被杀 |
| 仇人现身 | 复仇目标进入可感知范围 |
| 宗门危机 | 宗门遭攻、掌门陨落、资源断供 |

感知渠道可沿用和扩展现有信息传播系统：

- 亲眼看见：直接高可信。
- 关系传讯：师徒、同门、道侣、亲族优先知道。
- 宗门通知：宗门大比、宗门危机、集体秘境行动。
- 传闻传播：坊市、商会、口耳、城镇。
- 神识/空间感知：仇人现身、附近机缘。

## 6. 动态目标

动态目标是“这个事件对某个 NPC 意味着什么”。它不是常驻需求，而是在事件存在、NPC 知晓、且仍有窗口价值时产出。

### 6.1 ImmediateGoal

立即目标，适合当前就需要处理的突发事件：

- `escape_attack`：遭袭后逃命。
- `avenge_relative`：亲人、道侣、师傅被杀后追凶或立誓复仇。
- `rescue_disciple`：徒弟正在被追杀。
- `loot_fallen_master_now`：高手刚陨落，遗产窗口很短。

ImmediateGoal 可强打断当前计划，但仍要区分 Reaction 与 Utility。被攻击这类生死级事件走 Reaction；亲人被杀、遗产可抢这类走 DynamicGoal + InterruptPolicy。

### 6.2 PreparationGoal

准备目标，适合未来事件：

- `prepare_secret_realm`：秘境将在 90 天后开启。
- `prepare_tournament`：宗门大比将在 30 天后举行。
- `prepare_auction`：拍卖会将在 15 天后举行。
- `prepare_revenge`：知道仇人强于自己，先修炼、备丹、换法器。

准备行为包括：

- 补真气、补血、疗伤。
- 兑换或购买丹药。
- 修补或兑换法器。
- 收集情报。
- 找师傅、徒弟、同门结伴。
- 攒灵石、接任务换资源。
- 提前赶路到事件地点附近。

### 6.3 WindowGoal

窗口目标，适合事件已经进入可参与时间段：

- `enter_secret_realm`：秘境开启中。
- `join_tournament`：宗门大比报名或开赛。
- `attend_auction`：拍卖会进行中。
- `seize_treasure`：天材地宝可争夺。
- `loot_fallen_master`：高手遗产尚未被瓜分。

WindowGoal 过期后必须消失，并可写入记忆：错过秘境、错失机缘、拍卖失败等。

## 7. 事件生命周期

每个重大事件应具备时间视野：

```text
announced  预告期：可产生 PreparationGoal
preparing  准备期：NPC 可围绕事件调整当前安排
active     开启期：可产生 WindowGoal
resolved   结算期：写记忆、关系、收益、错过记录
expired    过期：不再产出目标
```

示例：秘境还有 90 天开启。

```text
WorldEvent(secret_realm, startsInDays=90)
  ↓
NPC 通过传闻得知
  ↓
DynamicGoalProvider 产出 prepare_secret_realm
  ↓
InterruptPolicy 评估是否出关
  ↓
如果值得：停止闭关，备药、攒灵石、赶路
  ↓
开启日：目标切换为 enter_secret_realm
```

## 8. 打断策略

`InterruptPolicy` 负责判断动态目标是否打断当前行为。

输出建议：

```text
interrupt_now      立即打断，清计划并重决策
after_step         当前 action 结算后重决策
keep_current_queue 加入待办，不打断当前计划
ignore             忽略
```

打断分数由以下因素组成：

| 因素 | 说明 |
| --- | --- |
| 事件价值 | 秘境资源、遗产价值、拍卖品、关系伤亡严重度 |
| 动机匹配 | 是否满足 NPC 当前缺口，如缺破境丹、缺法宝、缺延寿资源 |
| 关系强度 | 死者/遇险者是否是道侣、师傅、徒弟、亲人、恩人、同门 |
| 执念匹配 | 夺宝流、复仇流、长生流、传承流、夺权流 |
| 风险 | 秘境危险、遗产争夺者强弱、仇人战力 |
| 当前行为损失 | 闭关阶段、突破临近程度、任务是否即将完成 |
| 时间余量 | 距离开启还有多久，是否需要提前准备和赶路 |
| 性格 | 谨慎、勇敢、野心、忠诚、正义感 |
| 可达性 | 距离、行动速度、是否有资格参与 |

基础口径：

- 被攻击、濒死：Reaction 立即处理。
- 亲密关系死亡：高概率 `interrupt_now` 或 `after_step`。
- 秘境/大比/拍卖预告：通常是 `after_step` 或 `keep_current_queue`，但高度匹配当前缺口时可 `interrupt_now`。
- 闭关临近突破：除非生死、至亲死亡或极高价值机缘，否则不立即打断。
- 疗伤中：降低冒险类打断，除非事件直接关系生死或至亲。

## 9. 具体场景

### 9.1 闭关中得知秘境 90 天后开启

```text
NPC 当前：闭关中，totalProgress=0.92，缺破境助力
事件：秘境 90 天后开启，传闻有破境丹
判断：求道动机匹配高，时间足够准备
结果：打断或当前 step 后出关，开始 prepare_secret_realm
行为：备回血丹 → 兑换法器 → 接任务攒灵石 → 提前赶路
```

### 9.2 闭关中得知道侣被杀

```text
NPC 当前：闭关中
事件：道侣被杀，killerId 可定位
判断：关系强度极高，记忆强度高，复仇动机生成
结果：立即打断或当前 step 后重决策
行为：若敌弱则追凶，若敌强则 prepare_revenge
后果：可产生长期复仇执念
```

### 9.3 高手陨落，夺宝流 NPC 得知遗产地点

```text
NPC 当前：普通修炼
事件：高手陨落，遗产窗口 20 天
判断：求利/夺宝执念高度匹配，窗口短
结果：优先打断
行为：赶路 → 争夺遗产 → 结算收益/风险
```

### 9.4 谨慎养老 NPC 得知高风险秘境

```text
NPC 当前：归隐或疗伤
事件：秘境预告
判断：风险高，性格谨慎，动机不匹配
结果：忽略或只记录传闻
```

## 10. 与现有代码的接入边界

建议新增或演进以下单位：

| 单位 | 职责 |
| --- | --- |
| `WorldEvent` 数据结构 | 表达事件类型、位置、时间窗口、价值、风险、来源 |
| `EventAwareness` | 记录 NPC 知道哪些事件、可信度、来源、得知日期 |
| `DynamicGoalProvider` | 根据 NPC 状态和已知事件产出 Immediate/Preparation/Window Goal |
| `InterruptPolicy` | 判断动态目标是否打断当前计划 |
| `IntentService` | 合并基础需求、执念、关系、机会、动态目标 |
| `PlannerNode` | 消费打断请求，清计划并进入重决策 |
| `BehaviorSystem` | 保持计划与执行生命周期，不承担事件判断 |

现有 `npc-goals.js` 可承接第一版 `DynamicGoalProvider`，但长期看应拆成独立文件，避免额外目标越来越多导致文件膨胀。

## 11. 数据配置建议

新增事件目标配置时应保持数据驱动：

```json
{
  "id": "prepare_secret_realm",
  "eventType": "secret_realm",
  "phase": "announced",
  "basePriority": 65,
  "timeWindowDays": [15, 120],
  "motiveWeights": {
    "dao": 1.3,
    "profit": 1.2,
    "survival": 0.8
  },
  "requiredAwarenessConfidence": 0.4,
  "goalState": {
    "preparedForEvent": { "op": "eq", "value": true }
  }
}
```

第一版不需要一次实现所有字段，但字段口径要预留：事件类型、阶段、基础优先级、时间窗口、动机权重、关系权重、风险权重、目标状态。

## 12. 验证方式

遵循项目验证规则，不使用黄金指纹自证。验证应观察真实多种子长程模拟中的行为：

1. 秘境预告后，至少一部分匹配 NPC 会提前准备，而不是等开启日才动。
2. 谨慎或不匹配 NPC 不会被所有机会事件无脑打断。
3. 至亲/师徒/道侣伤亡能触发出关、复仇准备或追凶。
4. 准备目标能自然转为窗口目标，事件过期后目标消失。
5. 被动态目标打断的 NPC 后续能恢复正常行为，不永久卡在准备状态。
6. 日常修炼、任务、疗伤不因动态目标系统整体失衡。

建议新增验证脚本观察：

- 动态事件触发次数。
- NPC 知晓事件次数。
- PreparationGoal 产出次数。
- 打断结果分布：立即打断、step 后重决策、加入待办、忽略。
- 准备行为分布。
- 事件参与率与错过率。
- 打断后恢复正常行为的比例。

## 13. 分阶段落地

第一阶段：边界重构与动态目标最小闭环。

- 保留现有基础需求，先不大规模删表。
- 新增动态事件目标与打断策略。
- 接入秘境预告、宗门大比预告、高手陨落、关系死亡四类事件。
- 验证 NPC 会提前准备和合理出关。

第二阶段：需求表瘦身。

- 将完成任务、上交材料、猎妖资源等工作流需求迁移为目标解析或准备行为。
- 基础需求收敛到长期动机。
- 更新文档与数据规范。

第三阶段：人格与流派深化。

- 将求道、求利、求权、求情、求仇、求安等动机与 personality/obsession 深度绑定。
- 让同一事件对不同 NPC 产生明显不同决策。

## 14. 已确认决策

1. 采用 `DynamicGoalProvider + InterruptPolicy` 方案，而不是继续把事件塞进基础需求。
2. 未来事件也能影响当前计划，例如秘境 90 天后开启可以触发提前准备。
3. 准备目标可以打断闭关，但不是无条件打断。
4. 亲人、师傅、道侣被杀这类强关系事件可触发出关，并可能转化为长期执念。
5. 行为层只表达“做什么”，不表达“为什么做”。
