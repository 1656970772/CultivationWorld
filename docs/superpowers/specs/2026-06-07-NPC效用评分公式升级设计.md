# NPC 效用评分公式升级设计

> 最后更新：2026-06-07  
> 状态：待实施  
> 目标：把 NPC Utility 选目标层从裸乘法升级为更稳定的“门槛 + 基础欲望 + 加权效用 + 收益风险分离”公式。

## 背景

当前 `Goal.score()` 使用：

```text
score = (priority + Σdelta) × Π(modulator.mult) × Π(consideration)
```

这个公式符合 Utility AI 的基本思路，但在目标池变复杂后有两个主要问题：

1. 挂载更多 `consideration` 的目标会天然吃亏。
2. 任意一个软考量因素过低，都可能把高优先级目标压到几乎选不中。

NPC 决策已经拆成 `Reaction → Utility / Intent → GOAP → Job / Toil → Execution`。本次只调整 Utility 选目标层的评分聚合，不改变 GOAP 的路径规划职责，也不改变 Job/Toil 的执行流程。

## 决策

将目标评分升级为：

```text
base = clamp01((priority + Σdelta) / 100)

considerationMean =
  exp(Σ(weight_i × ln(max(value_i, floor_i))) / Σweight_i)

biasMult =
  clamp(Π(modulator.mult), minBiasMult, maxBiasMult)

rewardMult =
  1 + rewardWeight × expectedValue

riskMult =
  1 / (1 + riskWeight × goalRisk)

score =
  hardGate × 100 × base × considerationMean × biasMult × rewardMult × riskMult
```

### 字段含义

| 字段 | 含义 | 默认建议 |
|------|------|----------|
| `hardGate` | 真正的一票否决条件。当前阶段先固定为 `1`，后续再接目标有效性硬门槛。 | `1` |
| `base` | 基础欲望，沿用 `priority + Σdelta`，归一化后参与乘法。 | `0..1` |
| `considerationMean` | `consideration` 的加权几何平均，避免考量因素多的目标被系统性惩罚。 | `floor=0.05` |
| `biasMult` | 情绪、执念、上头等偏置乘子，带上下限防止叠爆。 | `0.25..3.0` |
| `rewardMult` | 期望收益吸引项，允许收益让目标超过基础分。 | `rewardWeight=0.5` |
| `riskMult` | 平滑风险惩罚，替代线性扣到极低的风险折扣。 | `riskWeight=1.0` |

## 实现范围

1. `apps/game/js/engine/abstract/goal.js`
   - 把 `_considerationProduct()` 改为加权几何平均。
   - 在 `score()` 中拆分 `deltaPriority` 与 `modulator.mult`，并对乘子做上下限。
   - 保持“无 consideration、无 modulator 时严格等于 priority”的兼容口径。

2. `apps/game/js/engine/abstract/consideration.js`
   - 使用已有 `weight` 字段。
   - 新增 `floor` 读取，缺省 `0.05`，用于软考量因素保底。

3. `apps/game/js/engine/npc/npc-utility.js`
   - 把 `goalRisk` 与 `expectedValue` 作为可解释调制项注入 Goal，而不是只藏在 `consideration` 派生值中。
   - 保留现有 `utilityConfig.enabled !== true` 时提前返回的默认关闭语义。

4. `apps/game/data/config/ai-config.json`
   - 在 `npc.utility` 下新增评分参数：`score.minBiasMult`、`score.maxBiasMult`、`score.defaultConsiderationFloor`、`score.rewardWeight`、`score.riskWeight`。

## 非目标

- 不改 GOAP 规划器。
- 不改 Job/Toil 执行系统。
- 不改具体 NPC 行为、任务、战斗或经济数值。
- 不启用 `utility.json enabled`、`reward.json enabled` 或 `reaction.enabled`；是否开启仍由后续平衡阶段决定。

## 测试与验证

单元验证：

- `Goal.score()` 在无调制、无考量因素时仍返回原始 `priority`。
- 多个 `consideration` 使用加权几何平均，不再因数量增加而线性相乘变小。
- `floor` 能防止软考量因素把目标压成 0。
- `biasMult` 按配置上下限夹取。
- `expectedValue` 与 `goalRisk` 分别生成收益放大与风险压制。

真实模拟验证：

- 默认关闭态跑现有测试，确认不会改变未启用 Utility 的默认路径。
- 激活态跑多种子长程模拟，观察目标选择分布、执念目标占比、夺宝/传承/养老/复仇是否仍能分化。
- 验证不得用固定摘要或哈希一致性作为正确性证据；只看真实行为统计和日志。

## 取舍

保留当前公式的优点是简单，但目标越复杂越容易被压低。改用加权几何平均后，公式稍复杂一些，但更适合长期扩展：目标可继续增加新的考量因素，不会因为“描述得更完整”而被评分系统惩罚。
