# 统一经济交易底座设计

> 最后更新：2026-06-07  
> 状态：设计已确认，待用户审阅  
> 类型：经济系统前置规格  
> 关联文档：`docs/superpowers/specs/2026-06-06-任务系统角色生涯任务网设计.md`、`docs/superpowers/specs/2026-06-07-三层关系底座全数据重构-design.md`、`docs/systems/item-covet.md`、`docs/systems/opportunity-system.md`、`docs/decisions/adr-015-faction-resource-and-promotion.md`、`docs/decisions/adr-025-item-covet-system.md`、`docs/decisions/adr-043-item-resource-unified-taxonomy.md`

## 一句话定义

统一经济交易底座不是普通商店系统，而是 **世界价值转移、托管、结算、债务和证据账本系统**：所有正式交易、任务契约、贡献兑换、材料上交、拍卖交割、私人交易、以物易物、罚没赔偿和违约债务都通过同一套资产读写、估价、托管、原子交割和分层账本规则处理。

## 背景

当前项目已经具备多条经济链路：

- NPC 和势力都有 `Inventory`，可持有灵石、材料、丹药、法宝、功法秘籍等可转移物品。
- 灵石已经作为 `currency` 物品统一进入 `apps/game/data/items/`，并可被交易和服用。
- 势力宏观资源 `low_spirit_stone`、`food`、`disciples` 已在 ADR-015 中统一以势力 `state` 为真相源，避免历史上的 `state` 与 `inventory` 双轨覆盖问题。
- NPC 侧已有材料捐献、贡献兑换、丹药服用、法宝装备、任务奖励和怀璧抢夺。
- 世界事件已有 `auction` 新闻和机会点入口，但拍卖仍是抽象收益，不具备真实拍品、竞价、托管和交割。

现有问题是：结算逻辑散落在 `quest-service`、`npc-economy`、`info-actions`、`faction-actions`、`world-rules` 等位置。任务契约、拍卖、押金、担保、罚没、退款、私下交易和债务追责需要同一套可解释的经济底座，否则后续任务系统会继续变成分散特例。

## 目标

1. 建立统一经济交易底座，覆盖物品、货币、组织点数、托管、交割、债务和交易账本。
2. 让正式交易场景默认可托管、可校验、可原子交割，成为稳定经济循环的基础。
3. 让私人交易保留违约、估价偏差、债务和后续追责空间。
4. 支持拍卖作为一等交易场景，并区分无玩家参与的抽象拍卖与玩家到场的完整拍卖。
5. 为任务系统提供任务报酬预存、押金、抵押、罚没、赔付、债务和流水证据。
6. 为 AI 提供经济信号，但不替代 AI、任务、事件和机会点系统生成交易动机。

## 非目标

以下内容不纳入本规格的第一版实现，但必须记录为后续经济系统待做：

- 不实现区域动态供需价格。
- 不实现完整真假货和伪装物系统。
- 不实现完整鉴宝职业系统。
- 不一次性重构所有产出与消耗系统。
- 不一次性实现店铺库存补货、宗门预算分配、炼丹炼器生产、物流押运和经济治理。
- 不让经济底座主动生成“谁想买卖”的目标。

## 总体架构

经济底座建议由以下核心模块组成：

```text
交易意图来源
  -> AI / 任务 / 动态事件 / 机会点 / 拍卖事件
  -> TransactionEngine
  -> AssetAdapter 读写资产
  -> PriceService 估价
  -> EscrowService 托管
  -> SettlementService 原子交割
  -> TransactionLedger 分层账本
  -> DebtService 长期债务
  -> EconomicSignalProvider 输出经济信号
```

设计原则：

```text
经济底座只负责价值校验、托管、交割、账本和信号。
交易动机由 AI、任务、事件、机会点和玩家行为产生。
```

## 资产范围

第一版纳入统一经济底座的资产：

| 资产 | 是否可交易 | 是否可抵押 | 说明 |
|------|------------|------------|------|
| 灵石 | 是 | 是 | `currency`，第一版以 `low_spirit_stone` 为基础计价单位 |
| 材料 | 是 | 是 | 灵草、矿材、妖丹、妖材、精血、灵果等 |
| 丹药 | 是 | 是 | 修炼丹、破境丹、疗伤丹等 |
| 法宝/装备 | 是 | 是 | 含已装备法宝，抵押或抢夺时需处理装备状态 |
| 功法秘籍 | 是 | 是 | 指可持有的秘籍物品，不是 NPC 当前修炼功法状态 |
| 贡献 | 否 | 否 | 只能由所属组织发放、扣除、冻结和兑换 |
| 战功 | 否 | 否 | 只能由所属军事/组织体系发放、扣除、冻结和兑换 |
| 宗门信用 | 否 | 否 | 组织评价信号，不作为私人交易资产 |

贡献、战功、宗门信用不是普通货币。NPC 不能私下赠送、买卖、借贷或抵押这些点数。

## 价格规则

第一版采用：

```text
成交参考价 = 物品基础 value × 交易场景倍率 × 私人交易修正
```

价格来源：

- **基础估值**：来自 `apps/game/data/items/*.json` 的 `value` 字段。
- **场景倍率**：坊市、宗门兑换、拍卖、急售、赔偿、罚没等不同场景可配置倍率。
- **私人交易修正**：根据关系、急迫程度、卖方缺钱、买方急需、报恩、宰客、仇怨、信息优势等调整。

第一版不做区域动态供需。  
第一版支持私人交易估价偏差，但不生成假货。物品都是真的，价格可能不公平。

## 交易场景

统一底座需要支持以下交易场景：

| 场景 | 第一版处理 |
|------|------------|
| 直接买卖 | 支持，正式场景默认安全，私人场景可违约 |
| 任务/契约结算 | 支持，含报酬预存、押金、罚没、赔偿和债务 |
| 贡献/战功兑换 | 支持，点数不可私下转让 |
| 材料上交 | 支持，材料进入组织库存，贡献由组织发放 |
| 宗门库存拨付 | 支持资产交割，权限由岗位/职权系统后续提供 |
| 势力贸易 | 支持结算接口，迁移时需谨慎处理势力 `state` 真相源 |
| 战争掠夺/赔款 | 支持资产转移和交易账本，具体战斗/战争动机由外部系统决定 |
| 以物易物 | 支持，但只用于私人交易、拍后交易、高阶交易、赔偿谈判、宗门特批 |
| 拍卖/寄拍 | 支持为一等场景，含托管、竞价、交割、流拍和财富曝光 |

## 正式交易与私人交易

交易安全采用分层规则：

```text
正式场景托管安全。
私人交易允许违约和估价偏差。
```

正式场景包括：

- 拍卖行。
- 坊市正规店铺。
- 宗门贡献/战功兑换。
- 任务堂备案任务。
- 悬赏阁。
- 有岗位授权的组织库存拨付。

私人或高风险场景包括：

- NPC 私人委托。
- 拍后私下交易。
- 荒野交易。
- 没有备案的炼丹/炼器代工。
- 黑活、秘密交易、人情交换。

正式交易默认通过托管和原子交割保证稳定。私人交易可以选择不用托管，也可以引入押金、见证人或第三方托管。

## 托管模型

托管采用混合模型：

```text
交易实例保存托管明细。
机构或见证人作为托管方。
```

托管用途包括：

- 任务报酬预存。
- 任务押金。
- 担保抵押物。
- 拍品寄拍。
- 拍卖入场保证金。
- 拍后私下交易托管。
- 赔偿和罚没待结算资产。

托管资产必须真实从原持有者库存移出，进入交易实例或机构托管账户。交易实例记录：

```js
EscrowEntry {
  itemId,
  quantity,
  nominalOwnerId,
  sourceEntityId,
  escrowHolderId,
  purpose,
  status
}
```

成功时划转，失败或流拍时返还，违约时罚没或赔付。

## 交易账本

统一底座保留交易账本：

```js
TransactionRecord {
  id,
  type,
  status,
  day,
  parties,
  assets,
  escrowRefs,
  debtRefs,
  visibility,
  source,
  evidence
}
```

交易类型包括：

- 购买。
- 出售。
- 以物易物。
- 贡献兑换。
- 战功兑换。
- 材料上交。
- 任务报酬。
- 押金。
- 托管。
- 退款。
- 罚没。
- 拍卖成交。
- 拍卖流拍。
- 赔偿。
- 债务偿还。

账本可见性采用分层账本：

```text
系统真相层：所有交易真实记录。
机构/公开层：拍卖行、任务堂、悬赏阁、宗门库房等只能查自己经手或公开登记的交易。
个人认知层：私人交易默认只有参与者、见证人和后续调查者知道。
```

交易流水是后续调查、审判、追责、声望、封禁、债务和任务链的证据来源。

## 债务模型

经济底座建立长期债务记录：

```js
DebtRecord {
  id,
  debtorId,
  creditorId,
  origin,
  dueDay,
  assetsDue,
  guarantorIds,
  collateralRefs,
  status,
  visibility
}
```

债务来源包括：

- 任务违约。
- 拍卖欠款。
- 私人借贷。
- 赔偿不足。
- 罚没不足。
- 担保人代偿。

债务状态：

```text
active / paid / overdue / forgiven / disputed / converted_to_quest
```

到期未还时，经济底座只产生逾期信号和事件，不直接替 NPC 做行为选择。后续由任务、关系、声望、岗位/职权、调查/审判系统生成追债、封禁、担保牵连、通缉或追杀。

## 拍卖系统

拍卖作为统一经济底座的一等交易场景。

### 模拟精度

拍卖采用玩家参与分层：

```text
玩家角色实际到场参与拍卖会
  -> 完整拍卖流程

玩家不在场
  -> 抽象竞价结算
```

重要 NPC 到场但玩家不在场时，仍走抽象结算。系统只记录谁参加、谁成交、成交价、财富曝光、结怨、后续截杀或私下交易等结果。

### 抽象拍卖

无玩家参与时：

- 根据参与者财力、需求、性格、关系、派系意图计算赢家和成交价。
- 不逐轮叫价。
- 成交、流拍、财富曝光、拍后交易机会和关系后果仍写入账本和事件。

### 完整拍卖

玩家到场时：

- 拍卖会按预展、入场、逐件拍品、叫价、弃拍、成交、流拍、交割、拍后交易推进。
- 所有在场 NPC 依据真实财力、需求、性格、关系、派系意图参与竞价。
- 竞价时校验支付能力或保证金，成交后走统一托管和交割。
- 高价竞拍、展示财力、寄拍珍宝会触发财富曝光和后续风险。

### 拍卖能力

拍卖第一版应支持：

- 拍品托管。
- 寄拍。
- 入场资格：请柬、身份、声望、展示财力或保证金。
- 竞价校验。
- 成交交割。
- 流拍返还。
- 拍后私下交易。
- 财富曝光和怀璧风险事件。

手续费、抽成、匿名寄拍、假货和复杂鉴宝先预留，不作为第一版必做。

## 经济信号

经济底座不生成交易动机，但提供 AI 可消费信号：

- 是否买得起。
- 是否可抵押。
- 是否有足够保证金。
- 是否能从组织库存兑换。
- 拍卖预估成交价。
- 私人交易价格是否偏离基准。
- 当前持有物是否高风险暴露。
- 是否有逾期债务。
- 是否被某机构交易封禁。
- 是否有托管资产待结算。

这些信号供 Utility、GOAP、Job/Toil、任务系统、关系系统和势力需求系统消费。

## 与既有系统边界

### 与物品系统

`Inventory` 仍是可持有实物的基础容器。经济底座通过 AssetAdapter 统一读写 NPC inventory、势力 inventory、势力 state 和组织点数。

### 与势力资源

势力宏观资源必须遵守 ADR-015：

```text
low_spirit_stone / food / disciples 以势力 state 为真相源。
```

迁移势力贸易、月俸、日常消耗等逻辑时，不能重新引入 `state` 与 `inventory` 双轨覆盖问题。

### 与关系系统

经济底座只记录交易事实、违约事实和债务事实。人情、背叛、信任、仇怨、通缉、封禁等社会后果通过关系事件和关系信号系统回写。

### 与任务系统

任务系统使用经济底座完成：

- 发布者支付能力校验。
- 报酬预存。
- 押金托管。
- 抵押担保。
- 成功奖励。
- 失败退款或部分结算。
- 违约罚没。
- 背叛赔付或债务。
- 交易流水作为调查证据。

### 与岗位/职权系统

经济底座不判断“谁有权动用组织资产”。岗位/职权系统后续提供授权，经济底座只执行授权后的资产校验和交割。

## 分阶段建议

| 阶段 | 目标 |
|------|------|
| 阶段 1 | 建立统一资产读写、估价、交易记录和经济信号接口 |
| 阶段 2 | 建立托管、押金、退款、罚没和债务模型 |
| 阶段 3 | 接入任务契约结算，覆盖报酬预存、押金、违约、赔偿和交易流水 |
| 阶段 4 | 迁移正式交易场景：贡献兑换、材料上交、悬赏阁奖励、坊市普通买卖 |
| 阶段 5 | 实现拍卖：无玩家抽象结算、玩家到场完整流程、寄拍、交割、流拍和财富曝光 |
| 阶段 6 | 接入高级风险与追责：私人交易估价偏差、债务逾期、交易封禁、调查/审判证据 |

## 后续经济系统待做

以下系统属于完整经济框架，但不进入本规格第一版范围：

1. **产出系统**
   - 灵脉、任务、采集、妖兽掉落、宗门产业、炼丹炼器、拍卖寄拍来源。

2. **消耗系统**
   - 修炼消耗、丹药消耗、装备损耗、宗门维护、月俸、任务预算、战争消耗、拍卖手续费。

3. **市场/店铺库存系统**
   - 坊市、拍卖行、宗门库房、悬赏阁、商会店铺分别有什么库存，库存如何补充、售罄和托管。

4. **组织财政与预算**
   - 宗门总库、任务堂预算、炼丹堂材料、执法堂罚没、拍卖行托管资产、岗位授权额度。

5. **生产与代工**
   - 炼丹、炼器、修复、鉴定、委托代工，材料转成成品，失败如何赔偿。

6. **费用与抽成**
   - 拍卖行抽成、寄拍手续费、坊市摊位费、宗门兑换折扣、任务堂备案费。

7. **物流与地点**
   - 物品所在机构、库房、角色身上；远程交易是否允许，押送和运输是否会被抢。

8. **经济治理**
   - 市场禁斗、拍卖行执法、宗门库房权限、交易封禁、黑活和私人交易追责。

## 用户已确认的设计决策

- 采用统一经济交易底座，不只服务任务系统。
- 第一版以实物、货币和组织点数为交易结算对象；人情、关系、声望仍由外部系统处理。
- 拍卖必须作为一等交易场景纳入底座。
- 无玩家参与的拍卖走抽象竞价结算；玩家实际到场参与时展开完整拍卖流程。
- 只有玩家角色实际到场才触发完整拍卖；重要 NPC 到场但玩家不在场仍抽象结算。
- 第一版价格采用基础 `value` 加场景倍率，动态供需不做。
- 私人交易可出现价格偏差，价格随卖方和交易双方状态变化；第一版不做假货。
- 正式交易场景默认托管安全，私人交易允许违约。
- 支持以物易物，但仅用于私人交易、拍后交易、高阶交易、赔偿谈判和宗门特批。
- 贡献、战功、宗门信用不可私下转让，也不能作为私人抵押物。
- 灵石、材料、丹药、法宝、功法秘籍等可交易道具可以抵押、担保、赔偿、托管和以物易物。
- 抵押、保证金、拍品寄拍和任务报酬预存必须真实转入托管账户。
- 托管采用混合模型：交易实例记录托管明细，机构或见证人作为托管方。
- 保留交易账本，每笔正式交易、托管、罚没、退款、拍卖成交和债务都可成为证据。
- 账本采用系统真相层、机构/公开层、个人认知层的分层可见性。
- 建立长期 `DebtRecord`，记录欠款、欠物、到期、担保、抵押和状态。
- 经济底座只结算并提供信号，不主动生成交易动机。
- 完整经济系统的产出、消耗、库存、财政、生产、费用、物流和治理另列待做。

## 验证要求

后续实施时，验证应观察真实行为和真实交易结果：

- NPC 和组织资产读写不会把势力 `state` 与 `inventory` 再次搞成双轨。
- 正式交易能校验资产、托管、交割并写入账本。
- 私人交易能出现价格偏差，并在违约时生成债务或争议信号。
- 押金、抵押、拍品寄拍会真实移出原库存并可返还或划转。
- 债务到期能产生逾期信号和后续任务/关系事件入口。
- 无玩家参与拍卖能抽象结算出赢家、成交价、流拍、财富曝光和账本。
- 玩家到场拍卖能逐拍品推进并通过同一套交割规则结算。
- 贡献和战功只能由组织发放、扣除、兑换，不能私下转让或抵押。
