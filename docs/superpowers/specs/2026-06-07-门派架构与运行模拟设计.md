# 门派架构与运行模拟设计

> 最后更新：2026-06-07  
> 类型：门派组织与运行模拟规格  
> 关联文档：`docs/superpowers/specs/2026-06-06-任务系统角色生涯任务网设计.md`、`docs/superpowers/specs/2026-06-07-统一经济交易底座设计.md`、`docs/superpowers/specs/2026-06-07-三层关系底座全数据重构-design.md`、`docs/decisions/adr-015-faction-resource-and-promotion.md`、`docs/世界观参考/宗门运行流程与制度平衡分析.md`

## 一句话定义

门派不是任务发布器，而是一个有管理层、堂口、弟子阶层、宗门背包、月俸开销、库存压力和内部利益冲突的组织生命体。任务、资源循环、弟子离宗、宗门衰败和后续玩家宗门经营都从这套门派架构中产生。

## 设计目标

1. 让宗门任务和悬赏任务有清晰来源，而不是凭空刷新。
2. 让宗门资源循环有实际压力：库存会消耗，库存不足会触发任务，长期断供会拖垮宗门。
3. 建立掌门、长老、堂口、堂主、堂口弟子和普通弟子的身份边界。
4. 为后续宗门经营、资源分配、堂口任命和内部斗争预留接口。
5. 保持第一版足够游戏化，不把堂口做成复杂行政模拟。

## 阶段总览

### 阶段 1：门派组织与资源压力闭环

- 建立门派组织架构：掌门、长老、堂口、弟子阶层。
- 建立堂口：执事堂、执法堂、炼丹堂、炼器堂、藏经阁/传功堂、杂物堂。
- 建立宗门背包初始库存。
- 建立月俸：灵石 + 丹药。
- 建立堂口额外俸禄。
- 库存低于安全线时自动生成任务需求。
- 长期欠发月俸或丹药、稳定度过低会导致弟子离宗。
- 弟子走完或死完时，宗门倒闭或灭门。

阶段 1 可以先把库存压力转成现有任务链可消费的任务需求；阶段 2 再把宗门任务、堂口任务和个人悬赏统一收束到完整任务列表与托管规则。

### 阶段 2：任务发布与悬赏托管

- 所有任务进入统一任务列表。
- 宗门和堂口任务给贡献，不托管奖励。
- 筑基期以上修士可发布个人悬赏。
- 个人悬赏奖励托管到宗门专门悬赏仓。
- 发布悬赏收固定灵石手续费，手续费归宗门。
- 任务取消时返还托管奖励，手续费不退。

### 阶段 3：堂口编制与任命

- 每堂少量固定编制。
- 普通堂口弟子至少筑基，有额外月俸。
- 堂主每堂 1 名，有更高额外月俸。
- 堂主不是管理层，没有资源分配权。
- 管理层只有掌门和长老。
- 掌门和长老可任命堂口成员、堂主，可调度弟子、发布任务、分配资源。

### 阶段 4：内部政治与勾心斗角

- 长老和掌门围绕资源分配、堂口任命、真传培养、高阶功法资格产生冲突。
- 弟子争堂口名额、争高阶功法试炼资格、争任务机会。
- 欠发月俸、丹药不足、资源分配偏袒会加剧关系恶化。
- 管理层可偏袒、打压、扶持、调度弟子。
- 这些冲突后续生成任务、悬赏、关系变化和事件链。

## 门派架构与数据边界

第一版门派仍以现有 `Faction` 为主体，不另起独立门派实体。门派新增一层组织配置和运行状态。

```text
门派
├── 管理层
│   ├── 掌门
│   └── 长老
├── 堂口
│   ├── 执事堂
│   ├── 执法堂
│   ├── 炼丹堂
│   ├── 炼器堂
│   ├── 藏经阁 / 传功堂
│   └── 杂物堂
├── 堂口编制
│   ├── 普通堂口弟子
│   └── 堂主
└── 弟子阶层
    ├── 杂役弟子
    ├── 外门弟子
    ├── 内门弟子
    ├── 核心弟子
    ├── 真传弟子
    ├── 长老
    └── 掌门
```

数据边界：

- **宗门背包**：保存可实际扣除、发放、兑换和回流的物品与资源。
- **宗门状态**：保存稳定度、欠发月俸、欠发丹药、成员存续等运行状态。
- **NPC 状态**：保存宗门身份、境界、堂口归属、是否堂主。
- **任务列表**：统一保存宗门任务、堂口任务和个人悬赏。
- **悬赏托管仓**：只保存个人悬赏奖励，不参与宗门普通消耗。

长老会只作为长老集合与议事语义，不作为第一版独立实体。实际管理权仍落在掌门和长老身上。

关键原则：

1. 弟子阶层不代表职责，只代表身份、待遇、权限和地位。
2. 堂口身份只代表额外月俸和归属标签，不提供效率加成。
3. 掌门和长老才是管理层，有调度弟子、分配资源、任命堂口成员和发布任务的权力。
4. 堂主不是管理层，只是堂口负责人身份，主要收益是更高月俸。
5. 宗门库存统一放宗门背包，不拆堂口库存。

## 堂口体系与任务来源

堂口不做复杂行政模拟，只做任务来源、兑换入口和身份标签。

### 执事堂

- 发布宗门任务。
- 发布或登记悬赏任务。
- 发布杂务、跑腿、送信任务。
- 发布普通采集任务。
- 发布事务性护送任务。
- 发布入门、月贡、晋升等考核任务。

### 执法堂

- 发布巡逻任务。
- 发布守卫任务。
- 维护宗门防务。
- 发布资源点护卫任务。
- 危机时发布支援任务。
- 执法堂人手不足时，向统一任务列表提交支援需求。

执法堂第一版不是审判系统，也不做复杂司法流程。

### 炼丹堂

- 发布丹材收集任务。
- 需求包括灵草、药材、妖兽内丹、特殊丹材。
- 提供丹药兑换。
- 不模拟炼丹协助和完整炼丹生产线。

### 炼器堂

- 发布炼器材料收集任务。
- 需求包括矿石、金属、妖兽材料、特殊炼器材料。
- 提供法器兑换。
- 不模拟炼器协助和完整锻造生产线。

### 藏经阁 / 传功堂

- 提供功法、术法、神通和秘法兑换。
- 高阶功法需要试炼任务。
- 试炼可包括战力试炼、贡献试炼、秘境试炼、师承认可等。
- 第一版至少支持“贡献 + 身份 + 境界 + 一个试炼任务”。

### 杂物堂

- 不主动发布杂项收集任务。
- 负责弟子上交任务、猎妖、采集、游历带回的杂物。
- 上交后给贡献，物品进入宗门背包。
- 提供符箓、阵盘、地图、令牌、普通消耗品等杂项兑换。

## 统一任务列表

所有任务进入统一任务列表。任务卡显示发布来源，不拆多个任务入口。

```json
{
  "issuerType": "hall",
  "issuerId": "alchemy_hall",
  "issuerName": "炼丹堂",
  "questBoard": "sect",
  "questKind": "sect_task"
}
```

```json
{
  "issuerType": "npc",
  "issuerId": "npc_xxx",
  "issuerName": "某筑基修士",
  "questBoard": "bounty",
  "questKind": "personal_bounty"
}
```

统一规则：

- 宗门任务、堂口任务、个人悬赏都显示在统一任务列表。
- 宗门和堂口任务主要给贡献。
- 个人悬赏给托管奖励。
- 其他堂口发布的任务也显示在同一列表，只标明来源堂口。

## 任务发布与结算规则

### 宗门 / 堂口任务

- 发布者：宗门、堂口。
- 来源：库存压力、月俸压力、丹药不足、防务压力、考核、秘境或传承需求。
- 奖励：贡献为主。
- 不需要提前托管奖励。
- 完成后，任务产物进入宗门背包或改善宗门状态。
- 总体原则：任务带给宗门的实际回报应大于发出的贡献。

### 个人悬赏

- 发布者：筑基期以上修士。
- 奖励：发布者自付。
- 奖励必须先托管到宗门专门悬赏仓。
- 发布时额外支付固定灵石手续费。
- 手续费归宗门，不退。
- 任务完成后，托管奖励发给完成者。
- 任务取消后，托管奖励返还发布者。

### 贡献点

贡献点是宗门内部信用积分。

- 只记录在个人身上。
- 不可交易。
- 不可私下转让。
- 不可托管。
- 不可抵押。
- 可用于兑换丹药、法器、功法、杂物、特殊资格。

### 悬赏托管仓

- 只保存个人悬赏奖励。
- 不参与宗门月俸、兑换、战争、维护等普通消耗。
- 是锁定奖励，不是宗门库存。

## 月俸、库存压力与宗门衰败

### 宗门开销

宗门每月有三类实际开销：

1. **身份月俸**
   - 按弟子身份和境界发放。
   - 内容包括灵石 + 丹药。
   - 身份决定基础档位。
   - 境界决定灵石和丹药品质与数量。

2. **堂口额外月俸**
   - 普通堂口弟子每月额外获得灵石。
   - 堂主每月获得更高额外灵石。
   - 是否额外给丹药可后续配置。

3. **基础维护费**
   - 每月统一扣灵石。
   - 表示宗门日常维护、建筑维护、护山大阵、灵脉维护。

贡献不是实际月度开销，兑换是库存变化，不纳入固定宗门开销。

### 初始库存

宗门初始背包必须配置足够支撑开局运转的库存：

- 灵石。
- 各境界常用丹药。
- 基础材料。
- 少量法器和杂项。

发月俸时实际从宗门背包扣除灵石和丹药。库存不足时，优先记录欠发，不凭空发放。

### 库存压力

库存低于安全线时生成任务需求：

- 灵石低于安全线：执事堂或管理层发布补充资源任务。
- 丹药低于安全线：炼丹堂发布丹材收集任务。
- 法器不足：炼器堂发布材料任务。
- 防务压力高：执法堂发布巡逻、守卫、支援任务。
- 杂物回收不足：杂物堂提高兑换需求或回收价值。

示例：

```json
{
  "resourceId": "pill_qi_gathering",
  "safeStock": 30,
  "criticalStock": 10,
  "issuerHall": "alchemy_hall",
  "questTemplate": "collect_alchemy_materials"
}
```

### 宗门衰败

- 长期欠发灵石月俸会导致稳定度下降，弟子可能离宗。
- 长期欠发丹药月俸会导致稳定度下降，弟子可能离宗。
- 宗门稳定度过低时，弟子可能离宗。
- 库存为 0 不直接倒闭，只通过欠发、稳定度、战力下降间接造成衰败。

倒闭或灭门条件：

- 宗门弟子全部离开。
- 宗门弟子全部死亡。
- 被其他宗门入侵后成员被杀完。

## 堂口编制与任命

堂口编制只做轻量身份，不做效率系统。

### 普通堂口弟子

- 至少筑基。
- 每堂少量固定名额。
- 每月有额外俸禄。
- 记录堂口归属。
- 不提供堂口效率加成。
- 不承担必须模拟的日常职责。

### 堂主

- 每堂 1 名。
- 优先金丹及以上。
- 小宗无人达标时，由本堂最高境界者代理。
- 每月有更高额外俸禄。
- 不是管理层。
- 没有资源分配权。
- 没有全宗调度权。

### 管理层

管理层只有：

- 长老。
- 掌门。

管理层权力：

- 发布任务。
- 调度弟子。
- 分配资源。
- 发起重大宗门行动。
- 任命堂口成员和堂主。

### 弟子身份层级

```text
杂役弟子
外门弟子
内门弟子
核心弟子
真传弟子
长老
掌门
```

弟子身份只影响：

- 月俸。
- 可接任务等级。
- 可兑换资源等级。
- 可参与活动或试炼资格。
- 宗门身份地位。

弟子没有固定职责，不负责调度。

## 内部政治蓝图

内部政治阶段不在阶段 1 完整实现，但设计上保留空间。

- 长老和掌门围绕资源分配、堂口任命、真传培养、高阶功法资格产生冲突。
- 弟子争堂口名额、争高阶功法试炼资格、争任务机会。
- 欠发月俸、丹药不足、资源分配偏袒会恶化关系。
- 管理层可偏袒、打压、扶持、调度弟子。
- 这些冲突后续生成任务、悬赏、关系变化和事件链。

内部政治不应绕开三层关系系统和现有 AI 决策链。它只提供压力、关系事件、任务来源和目标偏好，最终行为仍由现有 Reaction / Utility / GOAP / Job-Toil 链路裁决。

## 非目标

第一版不做以下内容：

- 堂口效率加成。
- 堂口独立库存。
- 堂口独立预算。
- 完整炼丹炼器生产线。
- 复杂审判系统。
- 复杂物流押运系统。
- 完整玩家宗门经营 UI。
- 每个堂口弟子的日常工作模拟。

## 验证原则

后续实施验证必须观察真实模拟行为，不使用指纹或哈希作为功能正确性的证据。重点观察：

- 宗门月俸是否真实扣除灵石和丹药。
- 库存不足是否产生对应任务需求。
- 宗门和堂口任务是否给贡献，并让产物回流宗门背包。
- 个人悬赏是否托管奖励并收取手续费。
- 欠发月俸、欠发丹药和稳定度过低是否会导致弟子离宗。
- 成员走完或死完时，宗门是否进入倒闭或灭门状态。

## 用户已确认的设计决策

- 完整蓝图采用“组织 + 库存压力闭环 + 内部政治雏形”，但分阶段实施。
- 第一版宗门架构采用掌门、长老会、堂口体系、弟子阶层。
- 堂口保留执事堂、执法堂、炼丹堂、炼器堂、藏经阁/传功堂、杂物堂。
- 后勤堂、库房、资源堂、外务堂不单独保留。
- 宗门库存统一记录在宗门背包，不拆堂口库存。
- 所有任务进入统一任务列表，只在任务卡上显示发布来源。
- 宗门和堂口任务给贡献，不托管奖励。
- 筑基期以上修士可以发布个人悬赏。
- 个人悬赏奖励必须托管到宗门专门仓库，发布时收固定灵石手续费。
- 贡献点是宗门内部信用积分，不可交易、转让、托管或抵押。
- 执法堂第一版主要负责巡逻、守卫、防务和危机支援，不做复杂司法系统。
- 炼丹堂只做材料收集任务和丹药兑换，不做炼丹协助。
- 炼器堂只做材料收集任务和法器兑换，不做炼器协助。
- 藏经阁/传功堂支持功法兑换和高阶功法试炼任务。
- 杂物堂主要做杂物回收和杂项道具兑换，不主动发布杂项收集任务。
- 堂口弟子至少筑基，每堂人数少，有额外月俸。
- 堂主是单独身份，有更高月俸，但不是管理层，没有资源分配权。
- 管理层只有掌门和长老。
- 弟子没有固定职责，只由身份影响月俸、任务权限、兑换权限、活动资格和地位。
- 宗门月俸包括灵石和丹药，根据身份和境界发放。
- 宗门实际固定开销只包括身份月俸、堂口额外月俸和基础维护费。
- 宗门初始背包必须配置支撑开局运转的库存。
- 库存不足会促使宗门发布任务，长期无人完成会导致宗门衰败。
- 宗门倒闭或灭门由成员全部离开、全部死亡或被入侵杀完触发。
