# 自动世界模拟与核心 NPC 池设计

> 最后更新：2026-05-27

## 背景

当前项目已经具备 `WorldEngine`、`DebugTimeline`、地图渲染、势力 AI、事件系统和 100 天引擎冒烟脚本。现阶段的世界模拟仍以玩家行动点触发 Tick 为主，`npcs.json` 中每个势力只有一名掌门，`resources.disciples` 只表示抽象弟子总数。

本阶段的目标是先把“主角不参与的世界”跑稳：世界可以自动推进、地图可以看到势力变化、日志可以追踪所有关键数值和决策概率，模拟结束后可生成分析报告，再根据报告迭代世界规则。

## 目标

第一阶段以稳定性为主，后续再逐步加入戏剧性和真实性。

- 支持无人主角的自动世界模拟。
- 保持自动模拟和未来玩家接入共用同一套 `WorldEngine`。
- 每轮模拟可以在地图上看到领地、战争和世界状态变化。
- 输出足够详细的结构化日志，覆盖输入数值、权重拆分、概率、随机值、命中区间和执行结果。
- 支持 100 天冒烟模拟和 1000 天压力模拟。
- 为每个势力加入少量具名核心 NPC，每个势力总数不超过 20 人。
- 让具名 NPC 参与继任、伤亡、事件对象和调试报告，但第一阶段不让每个 NPC 独立日常行动。

## 非目标

- 不在第一阶段实现全 NPC 独立 AI。
- 不在第一阶段追求战争、结盟、灭门等戏剧事件的高频爆发。
- 不把 `resources.disciples` 拆成全部具名人物。
- 不为自动模拟单独复制一套世界规则。
- 不在第一阶段重做地图渲染器或 UI 框架。

## 架构

自动模拟采用“同核不同入口”的结构：`WorldEngine` 继续作为唯一世界演化核心，主线程和命令行工具只负责驱动 Tick、展示地图、收集日志和生成报告。

```text
数据配置 apps/game/data/
        │
        ▼
WorldEngine（唯一世界演化核心）
        │
        ├── TickResult → GameManager → Renderer/UI → 地图可视化
        │
        ├── DebugTimeline → 调试面板 / 导出 JSON / 导出文本
        │
        └── SimulationReporter → 100 天 / 1000 天分析报告
```

### 运行时自动模拟

游戏 Web 端增加自动推进能力，但不改变玩家接入边界。

- 当前阶段可以隐藏或弱化玩家操作，把页面当成“世界观测台”。
- 自动模拟按钮或控制器向 Worker 发送 `MULTI_TICK` 或分批 `TICK`。
- 每个 Tick 结束后仍走现有 `handleTickResult()`，让地图和调试面板自然更新。
- 后续玩家接入时，玩家行动仍然只是在 Tick 之间插入输入，不需要替换世界演化核心。

### 离线模拟

命令行模拟工具复用 `WorldEngine`，用于快速跑 100 天和 1000 天。

- 读取 `apps/game/data/*.json`。
- 初始化 `WorldEngine`。
- 按指定天数推进 Tick。
- 导出结构化时间线。
- 汇总稳定性指标。
- 生成 Markdown 或 JSON 分析报告。

离线模拟不依赖 DOM，不访问 Canvas，不处理 UI。

## 核心 NPC 池

`resources.disciples` 继续表示宗门总体规模。`npcs.json` 只保存具名核心人物，作为世界叙事和规则调试中的关键对象。

### NPC 字段

```javascript
NPC {
  id: string,
  name: string,
  factionId: string,
  role: 'leader' | 'elder' | 'heir' | 'core_disciple' | 'officer' | 'general',
  rankId: string,
  personality: {
    ambition: number,
    caution: number,
    loyalty: number,
    diplomacy: number
  },
  alive: boolean
}
```

### 数量规则

每个势力的具名 NPC 数量包含掌门本人。

| 势力规模 | 判定依据 | 具名 NPC 数量 |
|----------|----------|---------------|
| 大势力 | `resources.disciples >= 500` | 12-18 人 |
| 中势力 | `resources.disciples >= 150` 且 `< 500` | 8-12 人 |
| 小势力 | `resources.disciples < 150` | 5-8 人 |

任何势力都不得超过 20 个具名 NPC。

### 第一阶段行为边界

- 只有掌门参与每日势力 AI 决策。
- 非掌门 NPC 不独立行动。
- 非掌门 NPC 可作为掌门继任候选。
- 非掌门 NPC 可作为事件伤亡、俘虏、叛逃、出使、历练等后续规则的对象。
- 调试日志和报告需要能按势力统计存活 NPC、死亡 NPC、掌门更替和候选继任池。

## 掌门继任

掌门死亡时，优先从本势力存活的具名 NPC 中选择继任者，而不是立即随机生成空降掌门。

继任优先级：

1. `heir`
2. `elder`
3. `general` / `officer`
4. `core_disciple`

同级候选使用 `loyalty`、`ranks.json` 的 `successionScore` 和固定角色优先级计算。第一阶段不增加威望字段，避免继任规则依赖尚不存在的数据。

如果势力没有任何存活候选，不再随机生成新掌门；该势力标记为 `destroyed`，领地转为无主地，并在日志中标记为“继承链断绝，势力覆灭”。

## 日志与报告

第一阶段的日志不是普通文本流水账，而是可分析的结构化数据。

每个随机或概率相关决策至少记录：

- 输入状态。
- 候选行动或候选结果。
- 权重拆分。
- 归一化概率。
- 随机值。
- 命中区间。
- 最终选择。
- 执行前后状态差异。

模拟报告至少包含：

- 运行天数、初始种子、是否异常退出。
- 每个势力的领地、资源、稳定度变化。
- 战争、扩张、结盟、世界状态、NPC 死亡、掌门更替次数。
- 长期 idle 或无效行动统计。
- 数值异常：`NaN`、负资源、稳定度越界、无效领地、断裂 NPC 引用。
- 地图异常：同一格多重归属、势力领地引用不存在地图格。
- 日志完整性异常：缺少概率、随机值、命中区间或执行结果。

## 验收标准

### 100 天冒烟模拟

- 模拟可以完整运行 100 天，无异常中断。
- 所有势力都有有效掌门。
- 所有掌门都引用存活 NPC。
- 地图领地归属没有断裂引用。
- 日志包含每日势力决策、世界状态、事件、稳定度和 NPC 状态。

### 1000 天压力模拟

- 模拟可以完整运行 1000 天，无异常中断。
- 稳定度始终保持在 0-100。
- 资源不出现 `NaN` 或非法负数。
- 具名 NPC 不超过每势力 20 人。
- 掌门继任可以持续工作，不因候选耗尽导致势力无主。
- 报告能指出异常、热点势力、过度扩张、长期无效行为和数值失衡。

## 未来玩家接入

玩家后续接入时，不改变世界演化核心。

- 玩家是世界中的一个输入源，不是 Tick 的替代品。
- 玩家行动、神识感知、事件选择可以继续通过 `GameManager` 和事件总线进入世界。
- 自动模拟模式可以保留为开发工具和调试看板。
- 同一份 `DebugTimeline` 同时服务玩家模式和无人模拟模式。
