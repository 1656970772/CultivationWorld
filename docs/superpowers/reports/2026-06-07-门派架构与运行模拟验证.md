# 门派架构与运行模拟验证

> 最后更新：2026-06-08

本报告记录门派组织、月俸、库存压力任务、个人悬赏托管、弟子离宗和宗门倒闭闭环的验证结果。验证只使用配置校验、测试断言和真实模拟行为，不使用哈希、指纹或关闭新机制作为正确性证据。

## 验证命令

```powershell
Set-Location -LiteralPath apps/game
node --check tools/simulate-analysis.mjs
node tools/test-data-manifest-load.mjs
node tools/test-game-data-validation.mjs
node tools/test-resource-registry.mjs
node tools/test-sect-operation-platform.mjs
node tools/test-sect-runtime-integration.mjs
node tools/test-job-config-load.mjs
node tools/test-economic-transaction-platform.mjs
node tools/test-economic-integration.mjs
node tools/test-quest-reward-economy.mjs
node tools/test-monster-hunt-job.mjs
node tools/simulate-analysis.mjs 300 --seed=2606071
node tools/simulate-analysis.mjs 300 --seed=2606072
node tools/simulate-analysis.mjs 300 --seed=2606073
```

结果：以上命令均通过，三组 300 天模拟均无运行时异常。

补充探针：使用真实 `WorldEngine`、真实 `SectBountyService` 和真实经济系统发布 1 条个人悬赏，验证 `simulate-analysis.mjs` 的个人悬赏托管过滤口径能识别 `purpose=personal_bounty_reward` / `source.type=personal_bounty_reward`，观察到 `escrowCount=1`、`locked=1`。

## 机制观察

| 项 | 观察到的真实行为 |
|----|------------------|
| 数据 manifest 与 strict validator | 当前配置通过 `test-data-manifest-load.mjs` 与 `test-game-data-validation.mjs`；门派配置缺失、非法引用、任务模板 metadata 缺失等负例由 strict validator 拦截。 |
| 门派月俸与维护费 | 三组 300 天模拟均产生 91 次门派月度运行结果；月俸、丹药俸禄和维护费均写入经济账本。 |
| 库存压力任务 | 三组模拟分别生成 32、35、33 个库存压力任务；最终仍有 26-28 个任务处于开放状态，说明低库存会真实回流到任务板。 |
| 个人悬赏托管 | 三组 300 天模拟未自然产生个人悬赏托管记录；`test-sect-operation-platform.mjs` 覆盖了个人悬赏发布、奖励托管、交付释放和不叠加普通奖励。 |
| 弟子离宗 | 三组模拟分别观察到 59、64、56 次离宗，来源于欠薪、库存压力或低稳定度触发的门派运行规则。 |
| 宗门倒闭 | 三组模拟均出现 1 个宗门因 `no_members` 倒闭，倒闭原因写入 faction state。 |
| 通用任务板接取与交付 | `test-sect-operation-platform.mjs` 验证 QuestService 从通用任务板接取宗门任务、写入 `activeBoardQuestId`、完成并交付，同时验证缺少交付处理器时不会错误重置状态。 |
| 经济、任务和斩妖闭环 | 经济交易、任务奖励和多日斩妖 Job 测试均通过；门派任务板与既有经济/任务/斩妖路径未出现回归。 |

## 结论

门派运行已进入默认正式流程：门派组织和运行数值来自配置，启动期 strict validator 能拦截缺失和错误引用，月度运行能在真实模拟中产生月俸账本、库存压力任务、离宗和倒闭行为。个人悬赏在三组 300 天模拟中未自然出现，但托管和交割闭环已由平台测试覆盖，后续若需要更高自然出现率，应调整个人悬赏发布权重和触发条件，而不是增加测试特例。
