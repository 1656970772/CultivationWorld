# ADR-002：数据编辑器升级为 Tauri 2 桌面工具

> 日期：2026-05-25
> 状态：已决定

## 背景

当前游戏运行时采用 ADR-001 确定的纯前端单体架构：`index.html` 作为游戏主入口，原生 JavaScript、Canvas 2D、Web Worker 与 IndexedDB 完成运行、渲染、演化和存档。该架构适合游戏本体的静态部署与快速迭代。

数据编辑器承担的是项目开发工具职责，需要批量读取、校验、修改并保存 `data/` 目录下的 JSON 文件。纯浏览器方案可以通过 File System Access API 做降级编辑，但在 Windows 桌面环境下仍存在目录授权体验不稳定、备份能力有限、文件写入和项目路径控制不足等问题。

## 决策

游戏运行时继续保留纯前端架构，不引入后端，不改变 `index.html` 的主入口职责。

数据编辑器升级为独立的 Tauri 2 + Rust 后端 + WebView2 桌面工具：前端继续复用现有 Web UI 和 ES Module 代码，Rust 后端负责项目目录解析、JSON 文件读写、保存前备份、路径安全校验和数据校验命令。

## 考虑的方案

### 方案一：Tauri 2 + Rust 后端 + WebView2（选定）

- 使用 Tauri 2 提供 Windows 桌面外壳。
- 使用 WebView2 承载现有编辑器 Web UI。
- 使用 Rust 命令处理本地文件系统、备份、原子写入和校验。
- 通过 `desktop-dist/` 打包编辑器前端资源。

### 方案二：Electron + Node.js 后端

- 使用 Chromium + Node.js 打包桌面工具。
- 文件系统能力成熟，前端生态接入成本低。
- 包体更大，运行时资源占用更高，并且会额外引入 Node/Electron 安全边界治理。

### 方案三：继续纯前端编辑器

- 继续依赖浏览器、File System Access API、下载导出等能力。
- 部署和维护最简单。
- 对项目目录、备份、批量保存和跨浏览器一致性的控制不足，不适合作为长期开发数据工具。

## 选择理由

1. 保持游戏主运行时简单：桌面编辑器是开发工具链，不替代游戏入口，也不改变浏览器可运行能力。
2. 更适合本地数据生产：Rust 后端可以稳定处理 `data/` 文件读写、备份目录和路径越界防护。
3. 包体与资源占用更克制：Tauri 复用系统 WebView2，通常比 Electron 更轻。
4. 前端复用成本低：现有 `data-editor.html` 与 `js/editor/` 可以继续作为 WebView UI，只增加 Tauri 适配层。
5. 安全边界更清晰：文件能力集中在白名单命令中，前端不直接获得任意文件系统权限。

## 风险

- Rust、Cargo、Tauri CLI 与 Windows WebView2 运行时成为桌面构建前置条件。
- Tauri 2 API、权限配置和打包配置需要单独维护。
- 前端需要同时支持 Tauri 环境和普通浏览器降级环境，存储适配层需要测试覆盖。
- Windows 文件路径、中文路径和备份目录写入可能暴露兼容性问题。

## 后果

- 游戏本体仍可通过静态文件运行和部署，架构边界不变。
- 数据编辑器获得桌面级项目打开、读取、保存、备份和校验能力。
- 仓库会新增 `package.json`、`src-tauri/`、`desktop-dist/` 和 `tools/build-tauri-frontend.mjs` 等桌面工具链文件。
- 后续涉及数据编辑器的功能，需要同时验证浏览器降级路径和 Tauri 桌面路径。
