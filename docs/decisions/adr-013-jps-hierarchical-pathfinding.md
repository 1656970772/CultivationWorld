# ADR-013：寻路优化（JPS 跳点搜索 + 分层 HPA*）

> 日期：2026-05-29（2026-05-29 增补 JPS+ 预处理）
> 状态：已接受 · 已实现并验证（无头基准 + 与标准 A* 对照测试通过）
> 关联：`docs/decisions/adr-006-spatial-movement-and-timed-actions.md`（实体空间移动）

## 背景

300×300 地图、约 150 个 NPC + 400 只妖兽逐 tick 移动。原寻路为标准 A*（`Map<"x,y", tile>` + 字符串 key + 数组线性取最小），CPU profiling 显示在批量模拟中 `computePath` 自耗时占比高达 **37.8%**，是最大单一瓶颈，导致 700 天模拟耗时约 273 秒（390ms/天），无法支撑"模拟 300 年"。

参考资料：知乎/B站分层 A*（HPA*）笔记、Botea 等 HPA* 论文、JPS / JPS+ 跳点搜索系列文章。

## 决策

针对"障碍稀疏（仅 ~1% 河流不可通行）、均匀代价为主（沼泽 cost=2 约占 2%）、地形静态"的特征，采用三层优化：

1. **整型网格位图 `GridGraph`**：引擎初始化时把地图压成两个 `Uint8Array`（`passable` + `cost`），坐标→线性索引 `y*width+x`，全程整型运算，消除字符串 key 与 `Map.get/set`、`split` 开销。地形静态，构建一次全实体共享只读；领地归属不影响通行性，故领地变化无需重建。

2. **JPS（跳点搜索）`jps.js`**：4 连通变体。沿直线"跳跃"跳过无决策意义的中间格，只把含强迫邻居的跳点入开放表。采用"垂直为主轴、行进中递归探测水平子分支"的非对称剪枝（保证完整性且避免无限递归）。作为有 `GridGraph` 时的默认寻路。

3. **分层 HPA* `hierarchical-graph.js`**：地图按 16×16 切簇，相邻簇边界连续可通行段取中点建过渡点，簇内过渡点间用 JPS 预算簇内边，构成抽象图。查询时起终点接入各自簇最近 K 个过渡点，在抽象图上跑 A*，再用 JPS 细化每段。仅在超远距离（曼哈顿 > 96 格）启用。

`computePath(from, to, tileIndex, options)` 对外签名不变：`options.graph` 传 `GridGraph` 启用 JPS，`options.hier` 传抽象图启用 HPA*，二者皆缺省时回退原标准 A*（向后兼容）。`MovementSystem` 另保留短距离（≤16 格）直线贪心 + 路径缓存。

4. **JPS+ 预处理 `jps-plus.js`**（增补，参考"最快速的寻路算法 JPS"中的位运算/预处理一节）：地图静态，故对每格、每方向预存 step 表——"沿该方向最远可走步数"（正数=到跳点距离，应入队；非正=到墙/边界距离，仅用于目标在直线段内的拦截）。运行时 `jump` 由"逐格 `isWalkable` 扫描"改为"O(1) 查表跳跃"，彻底消除 JPS 的主要热点。4 连通下需保持与基础 JPS 同样的非对称跳点语义，故按"水平跳点标记 → 左右 step → 垂直跳点标记（含'可向侧向 jump 到水平跳点'，用左右 step>0 判定）→ 上下 step"的四阶段顺序预处理。预处理结果挂在 `GridGraph.jpsPlus` 上，`jpsPath` 检测到即自动委托给 `jpsPlusPath`，所有调用方（含 HPA* 簇内边预处理）零改动受益。

## 验证

与标准 A* 对照（500 随机用例，300×300 真实地图）：

| 指标 | 标准 A* | JPS | JPS+ | HPA* |
|------|---------|-----|------|------|
| 单次耗时 | 21.1ms | 1.0ms（≈21×） | **0.34ms（≈62×）** | 17.3ms |
| 可达性 | 基准 | 100% 一致 | 与 JPS 完全一致 | 一致 |
| 路径代价 | 最优 | 偏差均值 ~2 格（沼泽所致，可忽略） | **与 JPS 逐格相同（偏差=0）** | near-optimal |

JPS+ 预处理仅约 **50ms**（一次性）。500 用例对照：JPS+ 与基础 JPS 可达性不一致 0、代价偏差总和 0、最大单次偏差 0，即输出**完全一致**，仅更快。

整体模拟：寻路自耗时占比由 **37.8%（标准 A*）→ ~16%（JPS）→ ~1.4%（JPS+，`jumpPlus`+`jpsPlusPath`+HPA*+直线贪心合计）**，寻路彻底退出瓶颈榜，GOAP 规划成为绝对主瓶颈（约 73%）。

## 后果

### 正面
- 单次寻路提速约 24 倍，寻路不再是模拟瓶颈。
- `GridGraph` 整型位图为后续寻路相关功能（视野、影响图）提供高效基础。
- HPA* 作为大地图扩展的基础设施保留。
- 对外 API 兼容，旧调用与玩家端寻路不受影响。

### 负面 / 权衡
- JPS 在非均匀代价格（沼泽）上非严格最优，偏差均值约 2 格，对模拟无实际影响。
- 本地图障碍稀疏，HPA* 边际收益有限（JPS 已足够快），故阈值设得较高，多数寻路走纯 JPS。
- 新增 `grid-graph.js` / `jps.js` / `hierarchical-graph.js` 三个模块，初始化多约 0.7–1.3s 预处理（HPA* 抽象图构建，一次性）。

## 后续

- **GOAP 规划现为绝对主瓶颈**（实测 `plan` 42.5% + `_stateKey` 12.5% + `_applyActionForward` 11.2% + `_actionApplicable` 4.7% + `getEffects` 4.4%，合计约 73%）。要支撑"模拟 300 年"，下一步应优化 GOAP：决策节流（非每 tick 重规划）、状态键/效果缓存、行为剪枝、规划深度上限等。寻路侧已无优化必要。
- **路径平滑（参考文章的第 5 项优化）评估后不采用**：本项目为 4 连通网格 + 逐格步进，经典视线平滑（string-pulling）会引入对角穿越，破坏 4 连通合法性（实体可斜穿障碍角），且 JPS 输出拐点本已最少。NPC 表现为格子移动，无平滑需求，故不实现以免引入错误。
