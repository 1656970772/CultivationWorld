# ADR-004：自动世界模拟与核心 NPC 池

> 日期：2026-05-27
> 状态：已决定

## 背景

当前游戏运行时已经采用纯前端架构，世界演化核心运行在 Worker 内的 `WorldEngine`。现有模拟主要由玩家行动点间接触发 Tick，`npcs.json` 中每个势力只有一名掌门，势力的弟子规模则由 `resources.disciples` 抽象表示。

为了先把世界本身做稳定，需要支持主角不参与的自动模拟：世界可持续运行，地图可见变化，日志能解释每次决策和概率结果，并能在 100 天与 1000 天模拟后生成分析报告。

## 决策

采用“同一世界核心，多种驱动入口”的架构：

- `WorldEngine` 继续作为唯一世界演化核心。
- 自动模拟、离线压力测试和未来玩家模式都复用同一套 Tick 流程。
- Web 端自动模拟通过 `GameManager` / Worker 消息驱动 Tick，并继续使用现有 Renderer 展示地图变化。
- 命令行模拟工具直接复用 `WorldEngine`，用于生成结构化日志和分析报告。

NPC 数据采用“双层人口模型”：

- `resources.disciples` 继续表示势力总体弟子规模，用于资源、战力和规模判断。
- `npcs.json` 保存少量具名核心 NPC，用于掌门继任、事件对象、伤亡、日志叙事和未来玩家交互。
- 每个势力具名 NPC 总数不超过 20 人。
- 第一阶段只有掌门参与每日势力 AI，其他 NPC 暂不独立行动。

## 考虑的方案

### 方案一：只增加 NPC 数据

只在 `npcs.json` 中补充人物，不调整自动模拟和报告架构。

该方案最快，但人物不会真正服务调试，也不能解决无人模拟、地图观测和稳定性报告问题。

### 方案二：自动模拟模式 + 核心 NPC 池（选定）

在保留现有 `WorldEngine` 的前提下，增加自动模拟入口、离线报告工具和少量具名 NPC。

该方案能让世界先稳定运行，同时为玩家接入、事件系统和更复杂 NPC 行为留下扩展空间。

### 方案三：全 NPC 独立 AI

每个具名 NPC 都拥有独立日常行动、目标和状态机。

该方案长期潜力最大，但第一阶段会显著增加调试复杂度，容易干扰稳定性验证。

## 选择理由

1. 保持低耦合：自动模拟只是新的驱动入口，不复制世界规则。
2. 保持数据驱动：NPC 扩展仍落在 `apps/game/data/entities/npcs.json`。
3. 保持未来兼容：玩家后续接入时可以作为输入源加入 Tick，而不是替换 Tick。
4. 控制复杂度：先让世界稳定，再逐步加入戏剧性、真实性和全 NPC 行为。
5. 提升可调试性：结构化日志和模拟报告可以让规则迭代基于证据进行。

## 后果

- `npcs.json` 需要从“掌门列表”扩展为“核心 NPC 池”。
- 掌门继任逻辑需要优先使用本势力存活 NPC 作为候选。
- 调试时间线需要覆盖 NPC 池状态、继任、死亡和日志完整性。
- 自动模拟模式会成为开发工具的一部分，即使玩家模式完成后也应保留。
- 100 天冒烟模拟和 1000 天压力模拟会成为世界规则调整前后的基础验证。
