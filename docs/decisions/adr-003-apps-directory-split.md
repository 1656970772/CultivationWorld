# ADR-003：按应用边界拆分 game 与 editor 目录

> 日期：2026-05-26
> 状态：已决定

## 背景

项目同时包含游戏 Web 运行时和 Win 端数据编辑器。早期实现中，`index.html`、`data-editor.html`、`css/`、`js/`、`data/`、`src-tauri/`、`tools/` 等目录都位于仓库根目录，资源管理器中显得分散，也容易混淆游戏源码、编辑器源码、构建输出和临时文件。

## 决策

采用 `apps/` 应用分层结构：

- `apps/game/` 存放纯网页游戏运行时。
- `apps/editor/` 存放 Win/Tauri 数据编辑器。
- 根目录只保留项目级规则、文档、记忆和隐藏配置。

游戏与编辑器保留独立入口、独立样式、独立脚本和独立默认数据目录。编辑器仍可通过 Tauri 后端打开外部项目根目录或外部 `data/` 目录。

## 考虑的方案

### 方案一：只删除临时文件

只清理日志、缓存和构建输出，不移动源码。

该方案风险最低，但根目录仍旧分散，不能解决应用边界不清的问题。

### 方案二：按 `apps/game` 与 `apps/editor` 拆分（选定）

把游戏和编辑器分别收拢到各自目录，保留现有技术栈和运行方式。

该方案能显著降低根目录噪音，同时不需要重写游戏或编辑器。

### 方案三：拆成两个完全独立仓库

游戏和编辑器分别维护独立 Git 仓库。

该方案边界最强，但当前项目仍处于快速迭代期，拆仓库会增加数据同步和文档维护成本。

## 选择理由

1. 视觉和职责边界清楚：根目录不再直接堆放运行时代码、编辑器代码和构建产物。
2. 保留当前可运行能力：游戏仍是纯网页端；编辑器仍是 Tauri + Rust。
3. 后续拆仓库更容易：`apps/game/` 和 `apps/editor/` 已经形成自然迁移单元。
4. 构建缓存可以保留：`apps/editor/node_modules/` 与 `apps/editor/src-tauri/target/` 可继续加速本地构建。

## 后果

- 文档和脚本中的路径需要以 `apps/game/` 或 `apps/editor/` 为准。
- 编辑器的默认模板数据位于 `apps/editor/data/`，游戏发行数据位于 `apps/game/data/`。
- 若编辑器需要修改游戏数据，应通过“打开项目”选择 `apps/game/` 或 `apps/game/data/`。
