## 结论摘要

我的判断：项目已经不是早期空壳，而是进入了“系统已复杂、需要治理和回归保护”的阶段。游戏运行时已经落地了 GOAP、GOBT、Utility、记忆/执念/情绪、JPS/JPS+、妖兽、宗门晋升等核心系统；但编辑器数据契约、文档同步、测试入口、版本管理和少数运行时边界存在明显问题。

最应该先做的事情有四件：

1. 先建立可追踪的工程基线：初始化或接入 Git，清理/忽略本地依赖与构建产物。
2. 先修编辑器数据契约：`apps/editor/js/editor/schema-registry.js` 已经指向 v2 分层数据，但 `apps/editor/data/` 仍是旧扁平数据，导致多项编辑器测试失败。
3. 先修游戏运行时两个高风险同步问题：玩家行动 Tick 结果可能被重复处理；领地坐标键的分隔符在引擎和主线程之间不一致。
4. 再做玩法迭代：当前 720 天晋升验证显示中低层晋升正常，但高层稀缺席位没有发生补位/挑战，需要作为平衡和叙事目标继续调参。

来源：系统架构文档确认游戏运行时是原生 JS + Canvas + Web Worker + IndexedDB，编辑器是 Tauri 2 + Rust + WebView2，且编辑器需要兼容游戏 `data/` 格式（`docs/architecture/system-overview.md:9`, `docs/architecture/system-overview.md:13`, `docs/architecture/system-overview.md:15`, `docs/architecture/system-overview.md:16`, `docs/architecture/system-overview.md:43`）。

## 当前项目状态

### 架构状态

事实：项目采用双应用边界。

- 游戏 Web 端入口是 `apps/game/index.html`，运行数据在 `apps/game/data/`。
- Win 端编辑器入口是 `apps/editor/data-editor.html`，Tauri/Rust 后端位于 `apps/editor/src-tauri/`。
- 文档明确要求游戏逻辑在 Worker 内运行，渲染层只读世界快照，编辑器只服务数据生产。

来源：`docs/architecture/file-structure.md:104`, `docs/architecture/file-structure.md:108`, `docs/architecture/file-structure.md:109`, `docs/architecture/file-structure.md:115`, `docs/architecture/file-structure.md:116`, `docs/architecture/file-structure.md:117`, `docs/architecture/file-structure.md:126`, `docs/architecture/file-structure.md:127`。

### 实现状态

事实：游戏侧核心实现已经很厚。`apps/game/js/engine/world/tick-manager.js` 约 80KB，`apps/game/js/engine/npc/npc-actions.js` 约 48KB，`apps/game/js/engine/npc/npc-entity.js` 约 31KB；这说明玩法规则已大量沉淀，但也说明职责开始拥挤。

我的判断：短期不要再把新玩法直接塞进 `TickManager` 或 `npc-actions`，应先拆出可测试的小系统，例如宗门定时活动、晋升、NPC 生育/道侣、事件日志、风险结算等。

### 数据状态

事实：当前游戏地图实际是 300×300，共 90,000 格；旧文档和部分编辑器测试仍写着 100×100，共 10,000 格。

来源：本地命令 `node -e` 读取 `apps/game/data/world/map.json` 得到 `{ width: 300, height: 300, tiles: 90000 }`；旧范围见 `docs/overview.md:24`、`docs/plans/implementation-plan.md:38`、`docs/plans/implementation-plan.md:55`、`docs/architecture/file-structure.md:142`。

我的判断：这是“文档/测试未同步”，不是游戏数据本身错误。需要把第一版范围、文件结构说明、编辑器测试中的 100×100 假设统一更新。

## 主要问题清单

### P0：当前目录不是 Git 仓库

事实：在根目录执行 `git status --short --branch` 返回 `fatal: not a git repository`。

影响：无法可靠区分用户修改、生成产物、测试产物和我的修改；也无法建立提交基线、分支策略或回滚点。

下一步：确认该目录就是目标项目后，初始化 Git 或接入已有远程仓库。建立首个基线提交前，应检查 `.gitignore` 是否覆盖 `node_modules/`、`src-tauri/target/`、`desktop-dist/`、`__pycache__/`、`artifacts/` 等目录。

来源：清理规则见 `docs/architecture/file-structure.md:156`, `docs/architecture/file-structure.md:160`, `docs/architecture/file-structure.md:161`, `docs/architecture/file-structure.md:162`, `docs/architecture/file-structure.md:163`。

### P0：编辑器数据结构与 schema/测试不一致

事实：schema 与测试都期待 v2 分层路径，例如 `data/entities/factions.json`、`data/world/map.json`；但编辑器模板数据仍是扁平 `data/factions.json`、`data/map.json`。

影响：编辑器页面初始化失败、校验测试失败、Tauri 前端构建测试失败，后续用编辑器生产数据不可靠。

下一步：推荐把 `apps/editor/data/` 迁移为与 `apps/game/data/` 兼容的 v2 分层结构，至少包含 schema 当前列出的数据集。迁移后运行：

### P1：玩家行动触发 Tick 时，结果可能被重复处理

事实：`worker.onmessage` 收到 `TICK_RESULT` 会调用 `handleTickResult(payload)`；`consumeActionPoints()` 又等待 `_requestTickAsync()` 的结果并再次调用 `handleTickResult(result)`。

影响：同一个 Tick 结果可能导致 UI、图谱、日志、自动存档、事件弹窗处理两次。世界引擎本身不一定推进两次，但主线程副作用有重复风险。

下一步：让 `_requestTickAsync()` 只负责等待，或让统一 `worker.onmessage` 分发并支持 Promise resolver，不要两处同时调用 `handleTickResult()`。修复前应加一个小型回归测试或浏览器日志计数，确认一次行动点归零只产生一次 `WORLD_TICK_COMPLETE`。

来源：`apps/game/js/core/game-manager.js:83`, `apps/game/js/core/game-manager.js:284`, `apps/game/js/core/game-manager.js:285`, `apps/game/js/core/game-manager.js:295`, `apps/game/js/core/game-manager.js:318`。

### P1：领地坐标键分隔符不一致

事实：引擎侧 `tileIndex` 使用 `x,y` 作为键，`TickManager` 多处按逗号解析；主线程 `_applyTerritoryChanges()` 却用下划线解析 `coordKey.split('_')`。

影响：从 Worker 快照回填主线程地图时，领地变化可能无法正确应用到 `tileGrid`，导致地图 UI 与引擎世界状态不一致。

下一步：统一坐标键格式，建议封装 `parseTileKey()` / `makeTileKey()`，并同时兼容旧 `_` 键一次性迁移。至少给 `_applyTerritoryChanges()` 增加逗号格式测试。

来源：`apps/game/js/core/game-manager.js:358`, `apps/game/js/core/game-manager.js:359`, `apps/game/js/core/game-manager.js:360`；引擎侧按逗号解析见 `apps/game/js/engine/world/tick-manager.js:496`, `apps/game/js/engine/world/tick-manager.js:644`, `apps/game/js/engine/world/tick-manager.js:876`。

### P1：`analyze:100` / `analyze:1000` 脚本参数无效

事实：`package.json` 写的是 `node tools/simulate-analysis.mjs --days=100`，但 `simulate-analysis.mjs` 只按位置参数读取 `process.argv[2]`。`parseInt('--days=100', 10)` 是 `NaN`，脚本会回退到默认 500 天。

影响：开发者以为跑了 100 天或 1000 天，实际可能都在跑默认 500 天，性能和报告判断会失真。

下一步：二选一：把脚本改成 `node tools/simulate-analysis.mjs 100` / `1000`；或给脚本补 `--days=` 参数解析。

来源：`apps/game/package.json:9`, `apps/game/package.json:10`, `apps/game/package.json:11`, `apps/game/tools/simulate-analysis.mjs:5`。

### P2：文档仍混杂旧范围和新实现

事实：概览与实施计划仍写 100×100、12 个势力等早期目标；实际本地验证显示游戏初始化为 18 势力、151 NPC、300×300 地图。

影响：新人或后续 Agent 会按旧目标改错方向，编辑器测试也会继承旧假设。

下一步：把文档拆成“历史计划”和“当前状态”。保留旧实施计划可以，但需要标注已过期或已完成阶段；新增当前里程碑页，记录真实地图规模、数据结构、已实现系统和待修问题。

来源：`docs/overview.md:24`, `docs/plans/implementation-plan.md:38`, `docs/plans/implementation-plan.md:55`, `docs/plans/implementation-plan.md:62`, `docs/architecture/file-structure.md:142`。

### P2：核心模块职责过重

事实：`TickManager`、`npc-actions`、`npc-entity` 文件体量显著偏大，且涵盖战斗、晋升、道侣、生育、月度考核、事件日志、移动、复仇等多个主题。

影响：继续迭代会增加回归风险；测试虽已覆盖关键算法，但模块级定位和变更半径会越来越难控制。

下一步：先不要做大重构。每修一个问题或新增一条玩法链，就顺手拆出一个边界清晰的服务类/模块，并补测试。建议优先拆 `promotion/sect-events`、`npc-social`、`location-events`、`risk-resolution`。

来源：模块职责原则见 `docs/architecture/file-structure.md:115`, `docs/architecture/file-structure.md:116`, `docs/architecture/file-structure.md:117`；文件体量来自本地 `Get-ChildItem apps/game/js -Recurse -File` 检查。

### P2：高层晋升/复仇/情绪涌现还偏弱

事实：720 天晋升验证里低层晋升和大比正常触发，但 `晋入稀缺顶层(elder/heir)` 为 0，`挑战上位(challenge_promote)` 为 0；100 天模拟中复仇 PvP 触发为 0，峰值愤怒和心魔为 0。

我的判断：这不一定是 bug，更像平衡和叙事触发条件偏保守。系统已经有结构，但还没有让世界足够频繁地产生“高层权力流动”和“恩怨叙事”。

下一步：先定义目标观测指标，例如 720 天内每个大宗门至少出现 1-3 次高层席位挑战或补位；1000 天内至少出现若干复仇链路。再调 `cultivation.json`、`obsession.json`、`emotion.json`、`social.json`、风险与事件参数。

来源：本地 `node tools/verify-promotion.mjs` 与 `node tools/simulate-analysis.mjs 100` 输出。
