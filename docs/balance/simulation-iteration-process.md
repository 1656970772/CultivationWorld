## 0. 什么叫"世界运行起来"

「世界活起来」不是一句感性目标，而是一组**可观测、可证伪**的指标。本流程把它定义为下列 KPI 同时成立：

| 维度 | 指标 | 健康判据（5000 天） | 失败信号 |
|------|------|---------------------|----------|
| **人口可持续** | 末态存活 NPC / 初始 NPC | ≥ 60%（≥90/151） | 单调衰减趋向空城 |
| | 出生 vs 死亡 | 出生 ≥ 死亡 × 0.5 | 出生 << 死亡 |
| | 性别比 | 0.5 ≤ 女/男 ≤ 2 | 单性别枯竭 |
| **权力流动** | 势力覆灭次数 | ≥ 1（有淘汰） | 全员 100% 存活 |
| | 高层席位补位/挑战 | ≥ 1 | elder/heir 流动为 0 |
| **叙事涌现** | 复仇 PvP 击杀 | ≥ 1 | 链路通但执念不行动 |
| | 执念分布种类 | ≥ 4 流派 | 单一流派 |
| | 历史峰值心魔/愤怒 | > 0 | 情绪系统静默 |
| **经济循环** | 妖兽死亡 / 任务完成 | 持续非零 | 资源链冷启动 |
| **健康度** | NPC GOAP 回退率 | < 50% | 规划层失效 |
| | 任务/猎妖失败率 | < 35% | NPC 自杀式接单 |

> 原则：**先定义指标，再调参**。没有指标的调参是盲调。

## 1. 标准迭代循环（每一轮 vN）

```
┌─────────────────────────────────────────────────────────┐
│ 1. 跑基线模拟      node tools/simulate-analysis.mjs 5000  │
│                    （全激活态环境变量见 §2）               │
│ 2. 采集            tools/report-data.js + 终端诊断日志     │
│ 3. 诊断            对照 §0 KPI 表，定位"最上游"瓶颈        │
│ 4. 归因            区分『参数问题』 vs 『结构问题』(§4)    │
│ 5. 备份            cp 待改文件 → docs/balance/backup/      │
│ 6. 最小调参        一次只动 1-3 个杠杆，避免混淆归因        │
│ 7. 重跑验证        同条件重跑，KPI 前后对比                │
│ 8. 记录            写 tuning-<date>-vN.md + result 文档    │
│ 9. 回归保护(按需)  功能性测试见 §8；黄金指纹默认不跑       │
└─────────────────────────────────────────────────────────┘
```

### 关键纪律

1. **一次只撬一类杠杆**：若同时改出生率 + 死亡率 + 招募，无法判断哪个起效。
2. **先上游后下游**：人口枯竭时先修"为什么不补人"，而不是先调复仇。死掉的 NPC 不会复仇。
3. **结构问题不要用参数硬调**：见 §4。把成功率从 0.2 调到 0.9 治不了"根本没有招募实体"的病。
4. **每轮必留回滚点**：改前备份到 `docs/balance/backup/pre-tuning-vN-*`。

## 2. 如何跑（命令与开关）

```powershell
cd apps/game

# 全激活态 5000 天（推荐基线：让 Utility/信息传播/关系 Goal 全开，世界规则充分运行）
$env:UTILITY_ACTIVE="1"; $env:INFO_ACTIVE="1"; $env:RELATIONSHIP_GOALS_ACTIVE="1"
node tools/simulate-analysis.mjs 5000

# 默认零漂移态（对照基线，验证「不开开关时的保守行为」）
node tools/simulate-analysis.mjs 5000
```

| 环境变量 | 作用 | 默认 |
|----------|------|------|
| `UTILITY_ACTIVE=1` | utility / reward / obsession.goalMult 开启（流派分化、期望收益） | off |
| `INFO_ACTIVE=1` | news / opportunities / reward / covet 开启（热点涌动、怀璧其罪） | off |
| `RELATIONSHIP_GOALS_ACTIVE=1/0` | 关系驱动 Goal/妖群/领地（强制开/关，对照实验） | 随 JSON |

> 这些开关**只在内存覆盖，不写回 JSON**，因此可安全用于对照实验。
> 产物 `tools/report-data.js` 被 `report.html` / `report-3000.html` 读取做可视化。

### 快速指标提取（无需打开 HTML）

```powershell
node -e "global.window={}; require('./tools/report-data.js'); const d=window.REPORT_DATA;
const c={}; for(const x of d.deaths) c[x.cause]=(c[x.cause]||0)+1;
console.log('末300死因', JSON.stringify(c));
console.log('出生', d.summary.totalBirths, '死亡', d.summary.totalDeaths);
console.log('存活NPC', d.summary.aliveNPCs, '男', d.summary.genderMale, '女', d.summary.genderFemale);
console.log('GOAP回退率', (d.goap.npcFb/(d.goap.npcOk+d.goap.npcFb)*100).toFixed(1)+'%');"
```

## 3. v3 基线诊断（2026-06-01，5000 天全激活态）

### 3.1 运行事实

- 引擎初始化：18 势力，151 NPC，用时 435s
- 末态：存活 NPC **38**（男 29 / 女 9），存活势力 18（**0 覆灭**）
- 出生 35，死亡 279，道侣 37 对
- 攻伐 18148，突破成功 27，复仇 PvP 击杀 **1**
- NPC GOAP 回退率 **63.3%**

### 3.2 人口曲线（核心病症）

| 天 | 50 | 500 | 1000 | 2000 | 3000 | 5000 |
|----|----|----|------|------|------|------|
| 存活 NPC | 136 | 90 | 74 | 57 | ~43 | **38** |

**单调衰减，无回升**。世界没有"活起来"，而是在缓慢"空城化"。这是当前最高优先级问题。

### 3.3 死因分布（末 300 例）

| 死因 | 占比 | 说明 |
|------|------|------|
| monster（妖兽） | ~53% | NPC 接取超出自身战力的猎妖任务 |
| quest / quest_hunt_failed | ~27% | 任务自杀式接单 |
| slain（PvP 致死） | ~9% | 攻战杀人（健康，叙事来源） |
| power_struggle | <1% | 夺权（偏少） |

### 3.4 根因归类

| 现象 | 类型 | 根因 |
|------|------|------|
| 人口只死不生 | **结构** | `faction_recruit` 只增加抽象 `disciples` 数值，**不生成新 NPCEntity 个体**；活跃个体只能靠道侣生育补充 |
| 生育杯水车薪 | 参数 + 结构 | 生育每 90 天检查、成功率 0.2，且受限于女性数量（9 个） |
| 性别失衡 9:29 | 参数 | 死亡未区分性别，但生育依赖女性，女性先枯竭形成正反馈 |
| 死亡过多（妖兽+任务 86%） | 结构 | NPC 接猎妖/任务前不自检战力（v2 已修 50%，仍不足） |
| 势力 0 覆灭 | 结构 | `winDefMin` 固定阈值，攻战杀 NPC 但抽象 disciples 数值仍高于阈值 |
| 复仇 PvP 仅 1 | 结构 | 复仇执念 goalState 在选目标层被其他需求压制（v2 已点名） |

> 结论：**"世界活不起来"的第一性瓶颈是人口结构** —— 没有可持续的「活跃个体」补充机制。这是结构问题，必须先解决，否则一切叙事（复仇、夺权、势力兴衰）都会随人口枯竭而停摆。

## 4. 参数问题 vs 结构问题（归因决策树）

```
观察到某 KPI 长期为 0 或单调恶化
        │
        ├─ 改 1 个参数后线性变化？  ──→ 参数问题，继续撬数据杠杆
        │
        └─ 改参数无响应 / 链路根本不存在？ ──→ 结构问题
                  │
                  ├─ 行为/实体根本没被创建？（如招募不生成个体）
                  ├─ precondition 死锁？（如 hasAdjacentEnemy 永不满足，见 v1）
                  └─ 链路断在某一环？（如复仇执念有了但不行动，见 v2）
                          │
                          └─ 需要改代码/数据契约，而非调数值
```

**历史教训**：
- v1 调数据杠杆（weight、阈值）→ 攻伐 0→775
- v2 修结构链路（attackEnemy 真正杀 NPC + 写记忆）→ 攻伐 775→10166、复仇执念 0→2
- v3（本轮）必须修**人口结构**，否则世界注定空城

- [ ] 末态存活 NPC 较基线 38 明显回升（目标 ≥ 60）

## 8. 黄金指纹的定位修订（2026-06-02）

> 背景：调了 v1~v6 + archetype 多轮后，仍感觉 NPC「呆」、且「怎么改都改回去」。
> 复盘发现，根因之一是**把『黄金指纹零漂移』当成了每次改动的硬关卡**，
> 而调行为的目标恰恰是**要改变行为**——两者直接冲突，导致真正能让 NPC 变活的开关被长期压着不敢动。

### 8.1 两类测试，定位完全不同

| 工具 | 它在测什么 | 调行为期该不该当硬关卡 |
|------|-----------|----------------------|
| `tools/test-goap-golden.mjs`（指纹 `5740e12a`） | GOAP 规划器对 600 个固定场景的输出 hash | **否**。这是为「纯重构（只搬代码不改行为）」设计的守护神，调行为期会变成绊脚石。 |
| `tools/refactor-baseline.mjs`（指纹 `4f9cf473` 等） | 200 天确定性模拟的世界终态 hash | **否**。同上，重构期专用。 |
| `tools/test-*.mjs`（复仇/师徒/关系/妖兽资源 等**功能性**测试） | 某条行为链「能不能跑通、有没有崩」 | **可保留为参考**：它们测的是「链路通不通」而非「行为是否一字不变」，调行为时仍有价值。 |

### 8.2 新规矩（自 2026-06-02 起）

1. **黄金指纹（`test-goap-golden.mjs` / `refactor-baseline.mjs`）默认不跑、不作为验收条件。**
   只有在以下两种情形、且**由人主动要求**时才跑：
   - 明确在做「纯重构」（拆类/改名/搬代码，承诺行为不变）——此时用指纹证明零漂移。
   - 怀疑某次改动**意外**影响了不该动的系统，想看「到底动了哪些场景」——把指纹当 diff 工具用。
2. **调行为期（让 NPC 不呆）不以指纹为准。** 指纹变了是**预期内**的，不是 bug。
   判断改动好坏的标准回归到 §0 的 KPI + 「人生回放」工具的体感，而非 hash 是否一致。
3. **文档里历史 ADR 中「黄金指纹零漂移」的表述全部保留**——它们是当时（重构期）的真实记录，不追溯修改。
   但**新的调行为迭代记录，不再写「黄金指纹零漂移」作为成功标志**。

### 8.3 那用什么兜底「改了不知道有没有改坏别的」？

不再用指纹硬卡，改用**更贴合「调行为」目标的护栏**：

- **KPI 对比**（§0 表）：改前改后跑同条件模拟，看人口/覆灭/叙事/健康度四维有没有意外恶化。
- **「人生回放」工具**（`report.html` →「🎬 人生回放」）：直接看几个代表 NPC 的一生，行为有没有变活、有没有变呆，肉眼可判。
- **功能性 `test-*.mjs`（按需）**：只在改动了对应系统（如改了复仇链就跑 `test-revenge.mjs`）时跑那一个，确认链路没崩——而不是全套指纹一起卡。

> 一句话：**指纹从「枷锁」降级为「按需的 diff 工具」。平时不跑，要用的时候你说一声。**
